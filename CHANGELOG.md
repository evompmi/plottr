## [2.4.0] - 2026-04-22

### Changed

- **UpSet + Boxplot — Configure step reordered so the data preview is the last tile.** Aequorin already follows this pattern (action tiles up top, preview at the bottom) and it reads much better — the user's eye lands on the decision they need to make instead of on 8 rows of parsed data they've already seen once on the upload step. Matched the pattern in `tools/upset.tsx` ConfigureStep (Sets-to-include → Intersection cutoff → Data preview; previously Preview was first) and `tools/boxplot/steps.tsx` ConfigureStep (ColumnRoleEditor → contextual warnings → Data preview; previously Preview was first). No behavioural change, purely reordering of sibling tiles so the primary action lives at the top of the scroll.

### Removed

- **All plot tools — bottom-of-step "Next →" / "← Back" navigation buttons removed.** Follow-up to the top-nav chevron + green-outline readiness cue: with the new affordance making "this step is reachable, click here to advance" unambiguous at the top of the page, the bottom-of-step duplicate navigation buttons (`Filter & Rename →`, `Output →`, `Plot →`, `← Filter`, `Continue to plot →`, `← Upload`, etc.) no longer earn their screen real estate. Removed 11 buttons across `tools/lineplot.tsx` (ConfigureStep "Plot →"), `tools/boxplot/steps.tsx` (3 clusters across ConfigureStep / FilterStep / OutputStep = 5 buttons), `tools/heatmap/index.tsx` (Configure step "← Upload" + "Continue to plot →"), `tools/aequorin/steps.tsx` (ConfigureStep "Plot →"), `tools/venn.tsx` (ConfigureStep "Plot →" — Venn's commit path was already replicated in `navigateStep`), and `tools/upset.tsx` (ConfigureStep "Plot →"). For UpSet, also lifted the configure-step's local `minDegree` / `maxDegree` state up to App as `pendingMinDegree` / `pendingMaxDegree` and taught `navigateStep` to patch `vis.minDegree` / `vis.maxDegree` via `updVis` when leaving Configure → Plot — closing the same silent-data-loss bug that affected the top-nav tab before the button was removed (previously only the bottom button committed the degree cutoff; the top "Plot" tab silently discarded it, mirroring the earlier Venn + UpSet set-selection bug). Also dropped the now-orphaned `setStep` / `canPlot` / `onCommit` props from the signatures and call-sites of `ConfigureStep` / `FilterStep` / `OutputStep` (boxplot + aequorin + lineplot + venn + upset). Two exceptions deliberately retained: boxplot's `← Configure` inside the plot-step error panel that appears only when the user reaches Plot without valid group+value columns (error recovery, not routine nav); and boxplot's "Switch to long pipeline" which isn't a nav button at all — it changes the data-format mode. Warning paragraphs near the removed buttons (`"Need at least two numeric columns to plot"`, `"Assign group + value columns"`, non-numeric value warning) were kept because they tell the user _what to change_, not just "can't advance yet". Updated `tests/components.test.js` StepNavBar assertion from 3 children to 5 (the 3 step buttons now interleave 2 chevron spans). All 750 tests, lint, typecheck, format:check, build pass.

### Changed

- **Step nav — chevron between each step button signals when the next step becomes reachable.** The top-nav step buttons now render a `❯` chevron (U+276F, 20 px) between adjacent pairs. Default colour is `var(--border-strong)` gray; the chevron animates to a new `var(--step-ready)` green (160 ms colour transition) only when it points forward to a reachable-but-not-yet-visited step — i.e., `nextStepIndex > currentStepIndex && canNavigate(nextStep)`. Past chevrons (pointing back at already-traversed steps) stay neutral so the "this way forward" cue is unambiguous. On top of that, step buttons that are reachable but not yet visited gain a 2-px `var(--step-ready)` halo rendered as `box-shadow: 0 0 0 2px` on top of the default 1-px border, so the outline is visible in both light and dark themes without causing a layout shift when state changes (border width stays constant). `--step-ready` is dedicated to this affordance (not reused from `--success-text`) because `--success-text` is a mint `#86efac` in dark mode so it can meet contrast on `--success-bg`, and that same mint glowed too brightly as a chrome accent; the new token is `#166534` in light (same dark green as before) and a muted `#4d8560` in dark. The step buttons themselves keep their click behaviour unchanged — the new affordances are purely visual cues that reduce the need to scan to the bottom of the current step panel to find out whether it's time to advance. Bottom "Next →" buttons in individual tool steps are untouched in this change; they can be phased out in a follow-up now that the top nav has a real readiness signal.

- **All plot tools — page header compacted: step buttons inline next to the tool name, vertical bar separators frame each segment.** The `PageHeader` + `StepNavBar` previously occupied two separate rows: the title (tool icon + name + subtitle) with the `PrefsPanel` right-aligned, then a full-width row of step buttons below. The title row never filled its horizontal space, and the step row duplicated the visual weight. Restructured `PageHeader` to accept a new `middle` slot and pull the step buttons into the header row: `[tool icon + title] | [step buttons] | [PrefsPanel]`, with 1-px vertical bars (`var(--border-strong)`) separating each segment. The subtitle now sits below the flex row in its own line. `StepNavBar` shed its `marginBottom: 20` (the shell no longer needs vertical spacing between two rows) and gained `flexWrap: "wrap"` so boxplot's 5-step chain wraps gracefully on narrow layouts. `PlotToolShell` passes the step bar as `middle` instead of rendering it separately below the header. Reclaims ~60 px of vertical space above the step content on every plot tool. No behavioural change to the step buttons themselves or to the `PrefsPanel` gating.

- **All plot tools — "Visual plot settings" button now only appears on the Plot step.** The `PrefsPanel` (auto-prefs load / save / reset) was rendered in the `PageHeader` right slot for every step, but it's only meaningful once the user is on the Plot step — there's nothing plot-related to save or reset from the Upload / Configure / Filter / Output steps. Gated the slot on `state.step === "plot"` inside `PlotToolShell` so the button is hidden until the user lands on the plot page. Removes a premature affordance and cleans up the header across the first 1–4 steps.

- **All plot tools — 120 ms opacity fade on step transitions.** Previously step changes (Upload → Configure → Plot → …) swapped the content region instantly, which felt abrupt — especially when the new step's content was visually dense (stats tables, sidebar controls, full chart). Added a `StepFade` wrapper inside `PlotToolShell` that watches `state.step` and runs an opacity 1 → 0 → 1 transition over 120 ms on change via a short `setTimeout` + inline CSS transition. Tracks the previous step in a ref so the initial mount doesn't fade in. Respects `prefers-reduced-motion: reduce` — for those users the transition is disabled entirely and opacity stays at 1. Applies uniformly to all seven plot tools because they all route step content through the shell's `children`. Nominal cost: 120 ms per transition; no layout reflow, no z-index juggling, no animation library added.

- **Upload step — separator-gate caption "Required before loading a file" removed.** The red caption appeared to the right of the separator dropdown while no separator was selected, duplicating the information the disabled drop zone below already conveys ("Pick a column separator above to enable file loading"). Removed the caption so the gate reads as one affordance instead of two stacked warnings.

- **Upload step — M-tier polish pass (collapsible "How to use" card, pill-chip example button).** Second batch of upload-page improvements from the UI audit, focused on information density and visual weight. The "How to use" info card under the drop zone was the single tallest block on step 1 — ~350–450 px of static content that most returning users have already read. Extracted the shared frame (outer border + gradient header + info-bg body) into a new `HowToCard` primitive in `shared-ui.js` with a button-as-header that toggles the body on click, rotates a `›` chevron 90°, and persists the open/closed choice in `localStorage` under `dv-howto-<tool>`. Defaults to open on first visit so existing walkthrough behaviour is preserved, then follows whatever the user last chose. Keyboard accessible natively (the toggle is a real `<button type="button">`), and `aria-expanded` + `aria-controls` let assistive tech announce state. All seven plot tools (`upset`, `scatter`, `lineplot`, `venn`, `boxplot/steps`, `heatmap/steps`, `aequorin/steps`) now consume `HowToCard` instead of duplicating the wrapper + header markup — ~40 lines of boilerplate removed per tool. The "No data handy? [Load example dataset →]" text-underline link at the bottom of the drop zone was re-styled as a `dv-btn dv-btn-secondary` pill with a "Try sample data:" prefix and a terser default label ("Load example →"), so it reads as an interactive action and not as a helper sentence. Ambient `HowToCard: FC<any>` added to `types/globals.d.ts` to match the other shared primitives. Build + lint + typecheck + format:check green; all 750 tests pass; no user-flow change (clicking the card header toggles, the inline how-to content is byte-identical).

- **Upload step — S-tier polish pass (microcopy, accessibility, token hygiene).** Bundled fix for eleven small issues flagged in a UI audit of every plot tool's step-1 upload page. Microcopy: the drop-zone headline is now "Drop CSV, TSV, or TXT — or click to browse" (was the more verbose "Drop your data file here, or click to browse"); the separator gate title is "Column separator" with a real `<label htmlFor>` binding (was "1. Choose your column separator:" — numbered for a step that isn't one); the oversized-file banner now appends "split the file or sample rows and try again" so the error is actionable; the `CommaFixBanner` second line drops the redundant "The data was corrected automatically." sentence (already implied by the heading). Consistency: added `exampleLabel="Plant biomass under drought / salt (3 genotypes × 3 treatments × 8 reps)"` to boxplot (was falling back to the generic "Load example dataset →"); removed the seven duplicate `<p>⚠ Max file size: 2 MB</p>` paragraphs that floated below `UploadPanel` in every tool and folded the 2 MB ceiling into `FileDropZone`'s existing hint slot — the limit now lives inside the affordance instead of as an afterthought paragraph (which also normalised the 5-tools-with-⚠ / 2-tools-without drift). Accessibility: `FileDropZone` is now keyboard-reachable (`role="button"`, `tabIndex=0`, Enter/Space handler, focus ring via `var(--accent-primary)`); all decorative emojis (📂 🚫 ⚠️ 🔄) get `aria-hidden="true"` so screen readers no longer announce "prohibited sign" / "open file folder"; `CommaFixBanner` gets `role="status"` and `ParseErrorBanner` + the size-error banner get `role="alert"` so state changes are announced; the disabled drop-zone placeholder's "Pick a column separator above" message replaces the previous "Select a column separator above to enable file loading" (terser, same meaning). Token hygiene: `FileDropZone`'s drag-over background was `rgba(100,143,255,0.06)` hard-coded and nearly invisible in dark mode on `--surface-sunken`; swapped to a new `--accent-primary-weak` theme variable (light: `rgba(147,158,174,0.12)`, dark: `rgba(58,68,84,0.28)`) that stays visible across both themes. No UX flow change — every keyboard and pointer path that worked before still works; parse-error banner location and how-to-card disclosure are untouched (those are M-tier items in the same audit and land separately).

## [2.3.0] - 2026-04-21

### Fixed

- **UpSet — column-id rotation threshold is now width-aware, not a magic `nCols > 10` cutoff.** Follow-up to the rotation fix below: the initial version hard-coded "rotate past 10 columns", but the real collision condition depends on `colW` and `idFontSize`, both of which can change independently of `nCols` (font-size preference, future layout tweaks, fewer columns but tighter spacing, etc.). Extracted the decision into `shouldRotateColumnIds(nCols, colW, idFontSize)` in `tools/upset/helpers.ts` — rotation now triggers as soon as the horizontal label width (`maxIdChars * idFontSize * 0.58 + 2 px gap`) exceeds the matrix column width. Added 5 regression tests in `tests/upset.test.js` covering: no-rotate when label fits, rotate when label overflows, colW-awareness (identical `nCols` reaches opposite decisions at different `colW`), font-size-awareness (same layout tips over as `idFontSize` grows), and the `nCols = 0` guard. Landing-page test counter bumped 671 → 676.
- **UpSet — column identifiers (`I1`, `I2`, …) tilt 90° once there are more than 10 intersections.** Horizontal labels collided and became unreadable past ~10 columns because the matrix cell width shrinks as the number of intersections grows. Above the threshold, each label rotates to read bottom-to-top (`transform="rotate(-90 cx y)"`, `textAnchor="end"`, `dominantBaseline="middle"`), occupying the same horizontal width as a single glyph regardless of how many digits the id needs. The bottom lane height and the legend Y-offset now derive from the longest rotated label so the `I# = intersection id …` caption and the SVG canvas (`VH`) grow to accommodate it without clipping. Behaviour is unchanged at ≤ 10 columns.
- **Venn — top "Plot" nav tab now respects pending set selection in Configure step.** Same bug pattern as the UpSet fix below: the `StepNavBar` "Plot" tab routed via `shell.setStep("plot")` directly, bypassing the `onCommit` path the bottom "Plot →" button used. Configure-step checkbox edits were silently discarded and the plot rendered against the previously-committed selection. `canNavigate("plot")` also reported readiness against the committed `setNames` rather than the pending one. Fixed with the same approach: a `navigateStep` wrapper commits `pendingSelection` (when it differs and has 2–3 sets, matching Venn's set-count gate) before calling `setStep`, and `canNavigate` now gates on `pendingSelection` while the user is in Configure. Audited the other four plot tools (Lineplot, Scatter, Heatmap, Aequorin, Boxplot) — none have the pattern: their step components write edits directly to App-level state via prop setters, so top-nav clicks don't discard anything.
- **UpSet — top "Plot" nav tab now respects pending set selection in Configure step.** Previously, the `StepNavBar` "Plot" tab routed via `shell.setStep("plot")` directly, bypassing the `onCommit` path that the bottom "Plot →" button used — so if the user toggled checkboxes in Configure and then clicked the top nav, the changes were silently discarded and the plot rendered against the previously-committed selection. `canNavigate("plot")` also reported readiness against the committed `displaySetNames`, not the pending one, so its enabled/disabled state was stale too. Added a `navigateStep` wrapper that commits `pendingSelection` when leaving Configure for Plot (only if it differs from the committed selection and has ≥ 2 sets), and updated `canNavigate` to gate on `pendingSelection` while the user is in the Configure step. The bottom "Plot →" button behaviour is unchanged.

### Fixed

- **Heatmap SVG export — exported files no longer render as a solid black canvas in Inkscape** — the root cause was the full-plot interaction overlay `<rect fill="transparent" width=plotW height=plotH>` in `HeatmapChart` (and the same attribute on the dendrogram hit rectangles): it was the last element inside the `<svg>` so it rendered on top of every cell, and Inkscape's SVG parser does not recognise the CSS keyword `"transparent"` as a valid value for the SVG `fill` attribute — it falls back to the default (black), painting a giant opaque black rect over the whole plot regardless of the actual cell fills underneath. Fix: switch every `fill="transparent"` in `tools/heatmap.tsx` (lines 564, 628, 1120) to the SVG-native `fill="none"` and add `pointerEvents="all"` so the rect still captures mouse events in the browser (without `pointer-events="all"`, an unfilled rect is hit-invisible by default per the SVG spec). Browser behaviour is preserved byte-for-byte; Inkscape now sees a non-fill rect and renders nothing on top of the cells, which is what was intended all along. Investigation also surfaced two latent cleanup items, applied at export time in a new `serializeSvgForExport(svgEl)` helper in `tools/shared.js` (routed through by both `downloadSvg` and `downloadPng`): (a) strips the root `<svg>` inline `style="max-width:100%;height:auto;display:block"` (responsive-layout sugar that has no meaning in a standalone file, and that some Inkscape builds mis-interpret on the root); (b) strips `shape-rendering="crispEdges"` on the cells group (redundant for export anyway since cells are already on integer pixel coords via `Math.round` in `cellX`/`cellY`, and older Inkscape/cairo versions have a known rasterisation quirk on grouped sub-pixel-height rects). The helper clones the node before mutating, so the live on-page chart is untouched.

- **Heatmap — detail (zoomed) plot now respects the column k-means inter-cluster gap and lines up with the main plot** — when the main heatmap was clustered with k-means on the column axis, the 10 px gaps between column clusters (introduced so the k groups read as discrete bands) made the main plot wider than `cellW * nCols`. The detail plot below it explicitly passed `colCluster={null}` to its `HeatmapChart`, so the gap math never ran in the detail: (1) the detail's SVG was narrower than the main's by the total inter-cluster gap, and (2) for any selection that started past at least one cluster boundary, the detail's columns landed left of their originating columns in the main, breaking the "stack the two plots and the columns line up" promise. Same problem in miniature for the row axis when row k-means was active. Three new opt-in props on `HeatmapChart` carry the fix: (1) `showClusterStrip` (default `true`) gates rendering of the col / row cluster strips and dendrograms — set to `false` by the detail because those strips belong to the main view; (2) `colGapStartPx` / `rowGapStartPx` (default 0) add a flat pixel offset to every cell's x / y, used by the detail to mirror the main's cumulative gap that sits to the LEFT of the selection's first column; (3) `basePlotW` / `basePlotH` are now AUTHORITATIVE when supplied (HeatmapChart no longer adds its own `totalColGap` on top), so the detail can pass `baseCellW * mainColCount + mainTotalColGap` and have its SVG match the main's exact width. `DetailView` now forwards `mainRowCluster` / `mainColCluster` (instead of `null`), computes the main's `totalColGap` and the cumulative gap before `cellOffsetCols` (replicating the same K_GAP loop), and passes both as `basePlotW` and `colGapStartPx`. Net effect: with column k-means on, the detail SVG is the same width as the main, the inter-cluster gap is drawn at the same boundaries inside the selection, and contiguous-selection columns sit at the exact pixel x of their originating main columns.

When more than one set is eligible, each now downloads as its own self-contained file named `<stem>_<slug>_stats.txt` / `.R`, with `<slug>` derived from the facet / subgroup label (Group Plot) or the x-value (Line Plot) via `svgSafeId`. Single-set and Aequorin (always one analysis) keep the existing single-file filename. Multi-file downloads fire with a small stagger so browsers don't drop them.
- **Stats-panel row hover uses a navy tint** — the per-row hover background in the Line Plot and Group Plot stats summary tables switches from the near-identical `--surface-sunken` to a new `--row-hover-bg` (soft navy `#d6deef` in light mode, indigo `#2a3766` in dark mode) so it's clearly visible that the row is clickable. Variable lives in `tools/theme.css` and is defined in all three theme blocks (light root, explicit dark, OS-preference dark fallback).
- **Aequorin stats tile redesign** — the `StatsTile` under the Ca²⁺ time-course plot is replaced by the same unified summary-table panel used by Line Plot and Group Plot (`AequorinStatsPanel`). Single row (Aequorin always has one set of groups) with test, statistic, p, stars; clicking expands it inline with slate-banner sections (Groups / Assumptions / Test / Post-hoc / Power) and exposes per-row **test override** (dropdown + "Use recommendation"). **↓ TXT** (decision-trace report) and **↓ R** (runnable reproduction) live at the panel header. Panel-level controls (Display on plot: Off / Letters / Brackets, Show ns, Print summary below plot) drive annotations on the inset barplot and the summary line under the chart, matching the vocabulary users already know from the other tools.
- **Group Plot stats tile redesign** — the per-facet / per-subgroup stack of `StatsTile`s is replaced by a single unified summary table (`BoxplotStatsPanel`) — one row per facet / subgroup with test, statistic, p, stars. A single pair of **↓ TXT** (stitched decision-trace report covering every facet / subgroup) + **↓ R** (runnable script reproducing every test) sits at the panel header, replacing the N pairs the old stack produced. Panel-level controls (Display on plot: Off / Letters / Brackets, Show ns, Print summary below plot) drive the chart's annotations and summary text for every set at once, removing the per-tile toggles that had no shared-chart meaning. Clicking a row expands it inline with slate-banner sections (Groups / Assumptions / Test / Post-hoc / Power) matching the Line Plot vocabulary, including per-row **test override** (dropdown + "Use recommendation"). The flat (no facet, no subgroup) case uses the same panel with a single row that auto-expands, so the visual vocabulary matches the multi-set case. Unlike Line Plot, no Benjamini-Hochberg adjustment is applied across facets / subgroups — tests are treated as independent experiments. The old side-by-side "trio" layout (plot + display tile + summary tile inside one wrapper per facet) is dropped; facet plots now sit in the grid above the unified stats panel.
- **Aequorin plot-page control panel restructured** — sections regrouped (Axes / Labels / Style / Summary barplot), X/Y inputs paired side-by-side, Labels + Style + barplot collapse by default, barplot Show toggle moved to the section header, a11y sweep for real `<label>` elements, Smooth slider gains a `pts` unit, debug column-grouping block removed. Bars now read colour directly from the main conditions — per-condition override dropped.
- **Aequorin control panels are collapsible** — four `PlotControls` sections wrapped in the same disclosure helper boxplot uses, so long condition lists don't overflow the sticky column. Other tools left as-is (fixed control counts).
- **Scatter regression Label position uses mini plot-frame icons** — each corner pill renders a tiny frame with a dot in the matching corner, replacing ↖↗↙↘ arrows.
- **Scatter regression and reference-line checkboxes → Off/On segmented toggles** — consistency pass covering Regression show, Dashed, Show equation, and per-reference-line Dashed.
- **Venn and Aequorin boolean checkboxes → Off/On segmented toggles** — Venn proportional-areas and Aequorin's four inset-barplot booleans now match the boxplot grid toggle.

### Fixed

- **Dark-mode Σ Baseline-corrected toggle dimmed** — the teal `--cta-plot-bg` (`#25a090`) read as glaringly bright against the dark chrome, especially next to the much quieter navy `Σ Raw` pill. Dropped to `#1d7a6d` so the two toggles now read as a matched pair in both themes.
- **Aequorin Y-axis auto-rescale rounds to 2 decimals** — when adjusting the X window, the recomputed `yMin`/`yMax` were writing 10+ decimal floats into the number inputs. Values are now rounded to `X.XX` so the controls stay legible.
- **Scatter regression default colour is a hex literal** — was leaking `var(--danger-text)` into SVG exports (CSS vars don't belong inside charts per CLAUDE.md).
- **Aequorin inset barplot CLD letters now display (#1)** — `InsetBarplot` was reading `annotations.letters`; `StatsTile` emits `labels`. Letters also moved to a fixed row under the plot frame, matching boxplot.
- **Group Plot enforces one `value` column** — configure step was letting users assign `value` to multiple columns; only the first drove the plot. Now demotes prior assignments to `filter`, matching the existing `group` rule.
- **Dark-mode dim fixes across chrome** — Molarity pills + mode cards, `.dv-disclosure` marker, Aequorin Σ Raw/Corrected pills, `.dv-btn-primary` CTA, Power `From my data`/chip toggles, Aequorin Combined/Faceted toggle, and misc. hardcoded hex leaks. All now use themed `--cta-*` / `--step-active-*` / `--accent-warning` variables that dim in dark mode.
- **`tinv` rewritten** — closed forms for df ≤ 2 and Newton-Raphson with Cornish-Fisher seeding for df ≥ 3. Matches R's `qt()` to machine precision even at extreme quantiles where the old pure-bisection `[-50, 50]` bracket silently clamped.
- **Zero-variance guards across stats** — `tTest`, `tukeyHSD`, `gamesHowell`, `oneWayANOVA`, `welchANOVA`, `leveneTest`, `cohenD`, `hedgesG` now return a populated `error` field instead of silently emitting NaN/Infinity on constant input; `shared-stats-tile.js` already surfaces it.
- **`hedgesG` exact gamma-ratio bias correction** — replaces the asymptotic shortcut. Matches R's `effectsize::hedges_g()` at small-n. Docs clarified on `fFromGroupMeans` (uses population-style SD) and `selectTest` (family-wise α inflation).
- **`sampleVariance` → Welford one-pass; `chi2inv` → Newton-Raphson** — Welford is stable under large offsets; `chi2inv` uses Wilson-Hilferty seeding with bisection fallback, matching R's `qchisq()`.
- **`compactLetterDisplay` and `kruskalWallis` degenerate-input guards** — CLD now skips NaN pairs instead of treating them as significant; Kruskal-Wallis all-tied case returns `NaN` + error instead of the misleading `H = 0, p = 1`.
- **Root-finder bracket expansion in `bisect`, `qtukey`, `powerAnova`** — fixed brackets were silently clamping to stale endpoints. All three now expand as needed; `bisect` refuses unbracketed targets.
