## [Unreleased]

### Fixed

- **Heatmap clustering — `hclust` no longer drops leaves when the distance matrix is all-NaN** — surfaced by the new fuzz harness: when every pairwise distance is NaN (easy to trigger with correlation distance on rows that share fewer than two finite values, or on rows with a constant series), the best-pair loop in `hclust` (`tools/stats.js`) never updates `bi`/`bj` because `NaN < Infinity` is false, the `if (bi < 0) break` exits with more than one active cluster, and the returned `tree`/`order` silently cover only one leaf. That made downstream `dendrogramLayout` and the main heatmap render an incomplete plot on degenerate inputs instead of showing a best-effort ordering. Fix: treat non-finite distances as unusable (`Number.isFinite(d)` check), and if no finite best pair exists among the active clusters, force-merge the two lowest-index ones at a sentinel height so the returned tree is guaranteed to cover every leaf. Added a pinning regression test in `tests/stats.test.js` ("all-NaN distance matrix still returns a complete leaf permutation").

- **Scatter — reference-line color picker no longer shows a broken "var(--danger-text)" value** — the `addRefLine` factory and the `resetAll` handler both seeded the line's `color` with the CSS-variable string `"var(--danger-text)"`, which is not a valid SVG `stroke` value nor a hex the `ColorInput` can parse. Result: the shared color swatch rendered empty and the in-SVG stroke fell back to the `|| "#444"` literal elsewhere in the render path. Replaced both with the hex literal `#dc2626` (the same red the Regression line defaults to, matching the regression `useState` init), so the picker now displays a valid swatch and the SVG stroke matches what the user sees.

### Changed

- **Render perf — three memoization wins on the chart hot paths** — three independent fixes that keep the SVG renderers from redoing the same work on every aesthetic tweak. (1) **Boxplot — KDE caching for violin / raincloud plots.** `kde()` is the dominant render-time cost on those plot styles (a Gaussian-density loop O(nPoints × n) per group), and `BoxplotChart` was invoking it twice per group per render — once to compute axis bounds, once to build the violin path geometry — every time the user dragged a slider, picked a colour, or otherwise touched the sidebar. Added a per-render `WeakMap` keyed on each group's `allValues` array reference (the upstream `useMemo` keeps that reference stable across aesthetic changes), so kde is computed once per data shape and reused across both call sites and across renders, until the underlying data actually changes. (2) **Heatmap — cells JSX wrapped in `useMemo`.** A 100 × 100 heatmap renders 10 000 `<rect>` elements; the chart re-renders on every mousemove (hover state, brush state, axis hover) and the cells block was being rebuilt from scratch each time even though it depends only on data + geometry + colour scale, none of which change during interaction. Memoized the cells array with explicit deps and also memoized `colGapOffsets` / `rowGapOffsets` (previously fresh `new Array(n)` each render, which would have invalidated the cells dep tuple). The hover tooltip and brush rectangles continue to re-render on their own; the cell grid stays cached. (3) **Sliders — `React.memo` on `SliderControl` with a custom comparator.** All sidebar sliders share a parent reducer, so dragging one was re-rendering every other slider in the same panel even though only one value had changed. Wrapped `SliderControl` in `React.memo` with a comparator that ignores the `onChange` prop (every call site passes an inline `(v) => updVis({...})`, which is a fresh function reference each render) — safe because every existing call site closes that arrow over a stable `useReducer` dispatch + a literal patch, with no captured mutable state that could go stale. Comment in `tools/shared-ui.js` flags the constraint for future contributors.

- **Scatter — "Ref lines" tile renamed to "Reference line" and moved next to "Regression line"** — the former "Ref lines" tile lived several slots below Regression (after the three aesthetic boxes and the Axes / Style sections), which split two conceptually-related overlays. Renamed to the more descriptive "Reference line" and moved to sit directly below Regression, so both line-overlay controls are adjacent.

- **Scatter — collapsible sidebar tiles with disclosure arrows** — the Plot step's controls sidebar matches the boxplot convention: `Point style`, `Style`, `Axes`, and `Ref lines` are now wrapped in a shared `ControlSection` that renders a disclosure arrow (the existing `.dv-disclosure` CSS) in its header. `Point style` and `Style` default to open (primary tweaks), `Axes` and `Ref lines` default to closed (power-user / on-demand). Variables (X/Y pickers), the three aesthetic `AesBox` tiles, and Regression keep their always-visible design because each already exposes its content through a built-in control (selects that reveal extras, an Off/On toggle for regression). Filters retains its pre-existing inline disclosure. Scrolls the opened section into view via the shared `scrollDisclosureIntoView` helper, same as boxplot and the filters panel.

- **Scatter — dropped the filename header from the controls sidebar** — the Plot step's sidebar started with a 32 px-tall panel showing `<filename> · N rows · M cols`, which duplicated information already visible in `DataPreview` on the Configure step and ate vertical space above the Actions panel (on short viewports, the controls immediately below it fell below the fold). Removed the panel; the file context stays reachable via the step-nav bar returning to Configure.

- **Settings gear button — bumped to a 40 × 40 px hit target** — the visual-plot-settings gear in `PageHeader` (all six plot tools) was 30 × 30 px with a 16 px glyph, which read as cramped next to the 40 px-tall theme toggle next to it. Enlarged to 40 × 40 px with a 20 px glyph so the two header buttons sit at matching heights and the gear is easier to hit with a trackpad tap.

### Removed

- **Venn — redundant "← Configure" back-button removed from the Plot step** — the top-of-plot secondary button duplicated the navigation already available via the step-nav bar, so it was taking vertical space (and a user-visible control slot) without adding functionality. Removed the button and its wrapper row; step navigation stays routed through the existing `StepNavBar`.

### Added

- **Fuzz test — seeded generator-based robustness harness for the heatmap data pipeline** — a new `npm run fuzz:heatmap` script (default 1000 iterations, configurable via `FUZZ_SEED` / `FUZZ_N` / `FUZZ_QUIET` env vars) that feeds thousands of plausibly-broken CSV/TSV strings through the full `parseWideMatrix → pairwiseDistance → hclust → kmeans` pipeline and fails on any uncaught exception, mis-shaped output, or leaf-order that isn't a valid permutation. 30 deterministic generators (`tests/fuzz/generators.js`) cover ragged rows, mixed delimiters, decimal commas, BOM/CRLF, quoted-field edge cases, duplicate headers / row labels, extreme numbers, NaN/Inf tokens, control characters, embedded null bytes, unicode labels, all-non-numeric cells, single-row / single-column matrices, and a 200 × 50 stress case. Each iteration uses a distinct seed derived from the top-level `FUZZ_SEED` so crashes are reproducible from a single integer; the first 5 repros print with the generator label, failure stage, exception message, and a 200-char excerpt of the input. A new `tests/helpers/heatmap-loader.js` loads `shared.js` + `stats.js` into a shared Node `vm` context with DOM stubs, mirroring the existing `shared-loader` pattern. Net-new: 2 files (~560 lines) + 1 `package.json` script entry. Intended as the template for per-tool fuzz suites in future PRs.

- **Fuzz tests — seeded robustness harnesses extended to all five remaining plot tools (boxplot, scatter, aequorin, lineplot, venn)** — five new `npm run fuzz:<tool>` scripts mirror the heatmap harness so every plot tool now has reproducible-from-a-seed coverage of its data pipeline, not just its render step. Each fuzz script feeds the shared corpus (the existing 30 CSV/TSV generators in `tests/fuzz/generators.js`, augmented with three new long-format generators — `genLongFormat`, `genLongFormatDegenerate`, `genLongWithExtremeGroups` — that produce the value/group/subgroup shape the long-format tools actually expect, including unicode and embedded-separator group names) through the tool's full pure-function pipeline and fails on any uncaught exception or shape-invariant violation. Per-tool: **boxplot** runs `parseRaw → bucket-by-group → quartiles + computeStats + kde → selectTest → chosen test → post-hoc + bhAdjust + assignBracketLevels` (caps at 8 groups so post-hoc combinatorics don't dominate runtime); **scatter** runs `parseRaw → numeric-pair extraction → linear regression → palette interpolation` and asserts r² ∈ [0,1] when valid; **aequorin** runs `parseWideMatrix → calibrate variants (linear / Hill / generalised, with bias toward degenerate parameters: Kr=0, n=0.01, n=100) → detectConditions in both pooling modes → smooth at random window widths` and asserts every cell is null-or-finite (no NaN/Infinity leaks); **lineplot** mirrors `computeSeries` and `computePerXStats` from `tools/lineplot.tsx` and validates per-x `pAdj` is null or in [0,1]; **venn** runs two independent loops — set-membership (`parseRaw → parseSetData → computeIntersections`) and geometry (random 2/3-circle layouts biased toward degenerate cases: tangency, full containment, near-coincident centres, zero radii) feeding `solveDistance → circleIntersectionPoints → buildRegionPaths → computeAllRegionAreas` — and asserts region count = (1<<n)-1, unique masks, and overlap area ∈ [0, π·min(r)²]. Loaders use three strategies depending on what the tool exposes: pure-JS slice (`tests/helpers/aequorin-loader.js` runs lines 1-177 of `aequorin.tsx`), mirror-inline helpers in the fuzz script (scatter regression, lineplot computeSeries / computePerXStats — kept in sync via source-reference comments because the originals carry TS generics), and esbuild `transformSync({ loader: 'tsx' })` on a slice (`tests/helpers/venn-loader.js` strips JSX from the first 977 lines of `venn.tsx`). All five harnesses validated clean across three seeds (1, 42, 987654321) at 5000 iterations each — 75,000 iterations per tool, no crashes. Net-new: 5 fuzz scripts + 5 loader helpers + 3 generators (~1500 lines) + 5 `package.json` script entries.

- **Actions panel — every download button now carries a descriptive tooltip explaining what the file contains** — the six plot tools (boxplot, aequorin, lineplot, scatter, venn, heatmap) share a single `ActionsPanel` (`tools/shared-ui.js:494-576`) that renders the `⬇ SVG` / `⬇ PNG` / extra-download chips plus a `↺ Start over` button. Until now every chip was a bare label — new users couldn't tell at a glance whether `CSV` meant the raw data, the plotted-and-filtered data, a summary-statistics table, or a membership matrix. Added native `title` tooltips: SVG / PNG / Start-over have fixed strings in `ActionsPanel` (same output everywhere), and each tool's extra-download entry now carries its own `title` describing what the file holds and how it's formatted — e.g. aequorin CSV is "calibrated [Ca²⁺] over time — one row per time-point, one column per sample", scatter CSV is "the filtered data table — only columns and rows currently drawn", lineplot Stats CSV is "per-x summary statistics (n, mean, SD, SEM, 95% CI) for every group", venn CSV is "membership matrix — one row per item, a 0/1 column for each active set", venn Regions is "one CSV per non-empty region (fires multiple saves — your browser may ask once to allow them)", heatmap CSV is "the plotted matrix — normalisation and row/column reordering applied", heatmap R script is "runnable R script that reproduces this plot with pheatmap". Boxplot has no extras, so only the shared SVG / PNG / Start-over tooltips apply there.

- **Venn — one-click "Regions" download dumps every non-empty region as its own CSV** — the Plot step's Actions panel gains a `⬇ Regions` chip next to the existing `⬇ CSV` (membership matrix) chip. Clicking it emits one CSV per non-empty region in the current diagram, each containing just the items that belong to that region. Filenames follow a readable convention — `<base>_venn_A.csv`, `<base>_venn_A_and_B.csv`, `<base>_venn_A_only.csv`, `<base>_venn_A_and_B_and_C.csv`, etc. — via a new `regionFilenamePart(label)` helper that rewrites `∩` → `and`, collapses whitespace to underscores, and strips any residual non-ASCII. The existing single-region download button in the Items panel uses the same helper, so naming is consistent whether the user pulls one region at a time or all at once. Empty regions are skipped (an empty CSV is noise, not a useful record). Modern browsers will prompt once to allow multiple downloads from a single user gesture — accepting is expected; a small `setTimeout` stagger (40 ms per file) avoids engine-internal races where a synchronous loop of `<a>.click()` silently drops later files. Saves users the previous "click region, click download, repeat N times" loop that got old fast on 3-set diagrams (7 regions).

- **Venn — files with more than 3 columns are now accepted, with a column picker in Configure** — previously `doParse` hard-rejected any file whose header row resolved to more than 3 non-empty set columns with a blocking error. Files from upstream pipelines often carry 4+ candidate columns (e.g. "genes up in stress_A / stress_B / control / qc_flag"), and users had to manually prune the file before loading. The tool now parses every column regardless of count and stores them as `allColumnNames` / `allColumnSets`; the Configure step gains a checkbox grid (one toggle per parsed column, each labelled with the column's item-count) that caps selection at 3 and requires at least 2 before the **Plot →** button enables. For files with 2–3 columns the picker is skipped entirely — `doParse` auto-commits the full column set and routes straight to the Plot step, so existing single- or two- or three-column workflows see zero UI change. The Plot step keeps its existing per-set toggle (the `activeSets` visibility filter) as a separate axis — "which of the file's columns to treat as sets" (committed in Configure) vs. "which of the committed sets to display right now" (toggled in Plot's sidebar). Navigating back to Configure after committing shows the prior selection pre-ticked so re-picking is a quick edit, not a restart; committing a different selection resets `selectedMask` and colour assignments.

- **All plot tools — user visual preferences persist across sessions, plus a portable Save / Load preset file** — the six plot tools (boxplot, aequorin, lineplot, scatter, venn, heatmap) now remember the user's render-side tweaks (colors, opacities, widths, sizes, toggles, palettes, etc.) between page loads. The page header gains a compact **Settings** strip on the right (same row as the tool title — kept out of the sidebar where control real estate is expensive) with three small buttons — **Save** (downloads `<tool>-settings.json` — a portable JSON snapshot of the full render state, title + axis labels included), **Load** (opens a file picker and applies the settings to the current plot), **Reset** (restores tool defaults and clears the stored preferences). The auto-persist layer deliberately _excludes_ plot/axis labels (`plotTitle`, `plotSubtitle`, `xAxisLabel` / `yAxisLabel` — auto-detected via a `/Title|Subtitle|AxisLabel$/` regex, not a hand-maintained allowlist), because those are dataset-specific and silently re-filling a title from the _previous_ experiment is more annoying than useful — labels still round-trip through the exported JSON file when the user explicitly opts in. Validation is defensive: tool-name mismatch on an imported file rejects with a one-line inline error, unknown keys are silently dropped, type-mismatched values fall back to defaults (so future rename PRs can't break older files). Storage lives under `dataviz-prefs-<toolName>` in `localStorage`, mirroring the existing `dataviz-theme` convention, with a 300 ms debounce so slider drags don't thrash storage. New shared module `tools/shared-prefs.js` (plain JS, loaded as a global `<script>` like the other `shared-*.js` files) exposes `loadAutoPrefs` / `saveAutoPrefs` / `exportPrefsFile` / `importPrefsFile` primitives plus a compact `PrefsPanel` React component; `PageHeader` (in `tools/shared-ui.js`) gained an optional `right` slot to host it alongside the title. 15 new unit tests in `tests/prefs.test.js` cover the validation / rejection / round-trip paths.

- **Heatmap — cell tooltips, interactive selection, and a linked detail panel** — hovering any cell in the main heatmap now shows a floating tooltip with the row label, column label, raw value, and (when normalisation is active) the plotted/normalised value. Three ways to build a selection: (1) drag a rectangular brush on the main grid; (2) click any internal branch of a row or column dendrogram to select that subtree's leaves; (3) in k-means mode, click a segment on the cluster-assignment strip to select every row/column sharing that cluster id. Selections are drawn as a white spotlight mask (60 % opacity on every non-selected cell) so the selected rows / columns stand out without any black outlines bleeding into the SVG export, and drive a second `HeatmapChart` rendered below the main one, showing only the selected rows × columns with its own SVG / PNG / CSV download buttons and a data table of the raw values (capped at 50 visible rows; CSV always has the full selection). A `Clear` button appears in the top-right of both plots when a selection is active. Selection indices point into the original unordered matrix, so re-clustering doesn't silently mis-highlight — any change to mode / k / distance / linkage / normalisation auto-clears the selection. The detail chart itself isn't interactive (tooltips only) to keep selection state single-sourced in the main view. Exporting the detail SVG gives a touch-up-ready figure of just the subset without having to go back to the source data and filter it by hand. The detail plot locks its column width and colour-bar length to the main plot and mirrors the main's dendrogram left-margin, so the two SVGs centre at the same width and the detail cells sit directly under the same columns in the main plot; cell _height_ in the detail is enlarged (14–28 px, scaled to the detail's own row count) so the selection actually reads as a zoom and row labels stop colliding when the names toggle is on.

- **Benchmark — clustering primitives cross-validated against R** — the R vs Toolbox comparison suite now exercises `pairwiseDistance` (Euclidean, Manhattan, 1 − Pearson) and `hclust` merge-heights (5 distance × linkage combinations) on a fixed-seed 100 × 15 Gaussian matrix. Comparison is done on sorted distance/height vectors so tie-handling differences don't spuriously fail the check while real numerical bugs still stand out. Currently: 8/8 new cases pass, max |Δ| ≈ 8e-7 — well inside the 5e-3 tolerance. K-means is intentionally left out of the benchmark: R's default (Hartigan-Wong, non-k-means++ init, incompatible RNG state) would make any labelled cross-check flaky noise; the deterministic-seed unit tests cover correctness on well-posed inputs.

- **Heatmap — visible gaps between k-means clusters** — when k-means is active on an axis, cells belonging to different clusters are now separated by a 10 px blank band (the cluster-assignment strip breaks at the same positions). Previously adjacent clusters read as one continuous colour strip with only the fill hue to signal the boundary; the new gap makes the k groups legible at a glance and matches the visual convention established by row/column dividers in tools like Morpheus and pheatmap's `gaps_row`/`gaps_col`. Gaps integrate with `plotW`/`plotH` so the colour bar, row labels, and selection overlays sit past the gapped grid's right / bottom edge; pointer hit-testing reports no hit when the cursor falls in an inter-cluster gap.

- **Heatmap — show row / column names toggles, and a built-in 500-row example** — tall matrices (hundreds of rows at a 2 px cell-height floor) left no vertical room for per-row labels, so row / column name visibility is now user-controllable via a pair of `Off / On` segmented buttons styled to match the existing clustering and grid toggles elsewhere in the app. Row names default to off (they can't be rendered legibly below ~12 px cells anyway); column names default to on. When labels are hidden the corresponding axis margin collapses to 0 so the plot doesn't reserve empty space. The app now ships with a deterministic 500-gene × 6-sample example (5 latent expression patterns × 100 genes each, `seededRandom(42)` so it's the same file on every reload) so the zoom / selection / clustering path can be exercised on a realistically tall matrix without having to hand-author one.

- **Heatmap — `NumberInput` stepper for k-means k and seed** — the two k-means numeric fields (per-axis `k`, 2–10; RNG seed, ≥ 1) now render through the shared `NumberInput` component, matching the +/- hold-to-repeat steppers used in Power, Aequorin, and Line Plot. Replaces the bare `<input type="number">` fields, which had inconsistent spacing with the segmented controls above them and no touch-friendly stepper buttons.

- **Heatmap — k-means (non-hierarchical) clustering** — the row and column clustering controls are now three-state (None / Hierarchical / K-means) instead of on-off booleans. Picking K-means on either axis runs k-means++-seeded k-means (with 8 restarts, inertia-minimising selection) on the normalised matrix (rows as observations, or the transpose for columns), reorders that axis so same-cluster rows/columns sit together (within a cluster, ordered by distance to centroid), and renders a thin Okabe-Ito-coloured cluster-assignment strip in place of the dendrogram. Per-axis `k` is exposed as a 2–10 spinner (default 3); a separate K-means seed input makes the clustering deterministic and lets users explore alternative initialisations. Hierarchical distance/linkage controls only surface when at least one axis is in Hierarchical mode; the K-means seed only surfaces when at least one axis is in K-means mode. The `.R` script export emits a `set.seed()` + `kmeans()` call (with `nstart = 8`) that reorders the matrix before handing it to `pheatmap`, and attaches a `factor(cluster)` annotation track on the k-means axis so the reproduced figure mirrors the browser one. New shared primitive `kmeans(matrix, k, { seed, maxIter, restarts })` in `tools/stats.js` — self-contained (local Park-Miller RNG, no cross-file dependency) so stats.js can be unit-tested in isolation. Handles NaN entries pairwise per feature, falls back gracefully on empty clusters by reseeding to the farthest row.

- **New Heatmap tool** — a browser-only tool for visualising numeric matrices (gene × sample expression tables, time × treatment readouts, metabolite × condition panels). Accepts wide-format CSV/TSV with a label column and label row; the numeric interior renders as a coloured grid. Optional hierarchical clustering can be enabled independently on rows and/or columns with three distance metrics (Euclidean, Manhattan, 1 − Pearson correlation) and three linkage methods (average / UPGMA, complete, single); clustered axes reorder the matrix and grow a dendrogram on top (columns) or the left (rows). Normalisation modes: none, Z-score by row, Z-score by column, log₂(x + 1). When a Z-score mode is active the tool auto-suggests the RdBu diverging palette anchored at 0; the default sequential palette is viridis. Twelve palettes total (viridis / plasma / magma / inferno / cividis / RdBu / bwr / RdYlBu / reds / blues / greens / spectral) — these also become available to Scatter via the shared module. **Borderless cells are the default** per a user preference for flat, gapless grids, with integer pixel-snapping and `shape-rendering="crispEdges"` so adjacent cells share a boundary and PNG exports don't open sub-pixel seams; a "No cell borders" checkbox reveals a `ColorInput` and width slider (0.25 – 2 px) when unchecked, for readers who want a gridded `pheatmap`-style look. Exports the named SVG groups (`cells`, `row-dendrogram`, `col-dendrogram`, `colorbar`, `row-labels`, `col-labels`, `title`, etc.) expected by Inkscape touch-up workflows, plus a CSV of the plotted values and a runnable `.R` script that reproduces the same clustering, normalisation, and colour scale via `pheatmap` (the raw matrix is embedded so reviewers can regenerate the figure without the original CSV). New shared primitives (`parseWideMatrix`, `pairwiseDistance`, `hclust`, `dendrogramLayout`, `COLOR_PALETTES`, `DIVERGING_PALETTES`, `interpolateColor`) live in `tools/shared.js` and `tools/stats.js` so future matrix tools can reuse them; `scatter.tsx` dropped its inline palette copies and now imports from the shared module, gaining the four new palettes for free.

### Fixed

- **Heatmap SVG export — exported files no longer render as a solid black canvas in Inkscape** — the root cause was the full-plot interaction overlay `<rect fill="transparent" width=plotW height=plotH>` in `HeatmapChart` (and the same attribute on the dendrogram hit rectangles): it was the last element inside the `<svg>` so it rendered on top of every cell, and Inkscape's SVG parser does not recognise the CSS keyword `"transparent"` as a valid value for the SVG `fill` attribute — it falls back to the default (black), painting a giant opaque black rect over the whole plot regardless of the actual cell fills underneath. Fix: switch every `fill="transparent"` in `tools/heatmap.tsx` (lines 564, 628, 1120) to the SVG-native `fill="none"` and add `pointerEvents="all"` so the rect still captures mouse events in the browser (without `pointer-events="all"`, an unfilled rect is hit-invisible by default per the SVG spec). Browser behaviour is preserved byte-for-byte; Inkscape now sees a non-fill rect and renders nothing on top of the cells, which is what was intended all along. Investigation also surfaced two latent cleanup items, applied at export time in a new `serializeSvgForExport(svgEl)` helper in `tools/shared.js` (routed through by both `downloadSvg` and `downloadPng`): (a) strips the root `<svg>` inline `style="max-width:100%;height:auto;display:block"` (responsive-layout sugar that has no meaning in a standalone file, and that some Inkscape builds mis-interpret on the root); (b) strips `shape-rendering="crispEdges"` on the cells group (redundant for export anyway since cells are already on integer pixel coords via `Math.round` in `cellX`/`cellY`, and older Inkscape/cairo versions have a known rasterisation quirk on grouped sub-pixel-height rects). The helper clones the node before mutating, so the live on-page chart is untouched.

- **Heatmap — detail (zoomed) plot now respects the column k-means inter-cluster gap and lines up with the main plot** — when the main heatmap was clustered with k-means on the column axis, the 10 px gaps between column clusters (introduced so the k groups read as discrete bands) made the main plot wider than `cellW * nCols`. The detail plot below it explicitly passed `colCluster={null}` to its `HeatmapChart`, so the gap math never ran in the detail: (1) the detail's SVG was narrower than the main's by the total inter-cluster gap, and (2) for any selection that started past at least one cluster boundary, the detail's columns landed left of their originating columns in the main, breaking the "stack the two plots and the columns line up" promise. Same problem in miniature for the row axis when row k-means was active. Three new opt-in props on `HeatmapChart` carry the fix: (1) `showClusterStrip` (default `true`) gates rendering of the col / row cluster strips and dendrograms — set to `false` by the detail because those strips belong to the main view; (2) `colGapStartPx` / `rowGapStartPx` (default 0) add a flat pixel offset to every cell's x / y, used by the detail to mirror the main's cumulative gap that sits to the LEFT of the selection's first column; (3) `basePlotW` / `basePlotH` are now AUTHORITATIVE when supplied (HeatmapChart no longer adds its own `totalColGap` on top), so the detail can pass `baseCellW * mainColCount + mainTotalColGap` and have its SVG match the main's exact width. `DetailView` now forwards `mainRowCluster` / `mainColCluster` (instead of `null`), computes the main's `totalColGap` and the cumulative gap before `cellOffsetCols` (replicating the same K_GAP loop), and passes both as `basePlotW` and `colGapStartPx`. Net effect: with column k-means on, the detail SVG is the same width as the main, the inter-cluster gap is drawn at the same boundaries inside the selection, and contiguous-selection columns sit at the exact pixel x of their originating main columns.

### Changed

- **`ColorInput` — typing a 3-character hex no longer auto-expands to 6 characters mid-keystroke** — the shared `ColorInput` component (used by every tool's colour pickers, cell-border swatch, group-colour editor, etc.) accepts both `#rgb` and `#rrggbb` forms. Previously, typing `#abc` into the text field would immediately commit the expanded `#aabbcc` to the parent state, which round-tripped through the component's `useEffect` sync and rewrote the text field to `#aabbcc` on the same keystroke — cursor jumping, typed shorthand forcibly lengthened. The fix is one line in `tools/shared-color-input.js`: guard the `useEffect` sync with `if (normalizeHexColor(text) !== value) setText(value)` so the effect only re-syncs when the external value is genuinely different from the user's current text (i.e. from a palette preset, theme reset, or external state change). Self-commits where the entered text already normalises to the new value no longer clobber what the user typed. Three-character hex still commits a valid expanded colour to the parent (so the `<input type="color">` and downstream SVG render see a proper `#rrggbb`); only the displayed text field keeps the user's original form.

- **Stats-tile checkboxes (`Display on plot`, `Print summary below plot`, `Show ns`) now use the slate `accentColor` matching the rest of the app chrome** — the three controls in the per-tool `StatsTile` (boxplot + aequorin) rendered as bare `<input type="checkbox">` with no `accentColor` style, so the browser's default UA blue ticked through and clashed with the soft slate accent everywhere else (selectors, primary buttons, slider tracks). ### Fixed

- **Power tool ANOVA n-solver jumped to 100000** for large effect sizes at k ≥ 6 — root cause was `ncf_sf` truncating its Poisson mixture at a fixed ±500-step window. Now widens to ±8σ with a `pUp < 1e-14` early exit and a normal-approx short-circuit for λ > 1000.
- **`gammainc` silently wrong at large `a`** — series loop had a fixed 200-iter cap; at a ≈ 10000 it needs ~780. Scaled the cap with √a. Propagated into `chi2cdf` and large-df F-tails.
- **`ncchi2cdf` truncated Poisson mixture** — same fix as `ncf_sf`: √(halfLam) window + early exit + normal-approx far tail.
