## [Unreleased]

### Added

- **Horizontal orientation for Group Plot** — a "Horizontal orientation" checkbox appears below the Plot style dropdown in the style controls panel. When checked, all four plot styles (box, violin, raincloud, bar) render with groups on the y-axis and values on the x-axis (like ggplot2's `coord_flip()`). Grid lines, axes, tick labels, CLD letters, significance brackets, and composition pies all rotate to match — pies sit to the left of each group label in horizontal mode. The toggle works in both non-faceted and faceted views. Default is unchecked (vertical, matching the existing behaviour).
- **Subgroup-by for Group Plot** — a "Subgroup by" dropdown in the style controls panel lets users partition groups into subgroups by a categorical column. All groups render in a single plot, separated by dashed vertical lines (horizontal in hz mode), with subgroup labels displayed inside the plot frame above statistics annotations. Statistics (assumption checks, main test, post-hoc, CLD letters, significance brackets) are computed independently within each subgroup — each subgroup gets its own collapsible StatsTile panel and its own annotation layer. This is mutually exclusive with faceting: selecting a subgroup column disables the facet selector and vice versa. Supports all four plot styles and horizontal orientation from day one.
- **Group Plot configure step requires group + value columns** — the "Filter & Rename →" button is now disabled with a warning banner when either a group or value column is missing from the role assignments, preventing users from reaching an empty plot.
- **R-script export from the Statistics display tile** — Group Plot and Aequorin now show a second download chip (`⬇ R`) next to the existing `⬇ TXT` report button. Clicking it emits a runnable `.R` script that embeds the current groups as an inline long-format `data.frame` and reproduces every test the tile just ran: per-group `shapiro.test`, `car::leveneTest(..., center = "median")`, the correct main test (`t.test` Student or Welch, `wilcox.test`, `aov + summary`, `oneway.test(var.equal = FALSE)`, or `kruskal.test`), and the correct post-hoc (`TukeyHSD(aov(...))`, `rstatix::games_howell_test`, or `rstatix::dunn_test(p.adjust.method = "BH")`). The header block lists every required package in a single `install.packages(c(...))` comment, embeds the toolbox's decision-tree rationale as a trailing comment, and — crucially — the generated script only calls R tests that are already cross-validated against the toolbox's JS implementations by the benchmark suite, so running the script reproduces what the tile shows to benchmark tolerance. This directly closes the "no reproducibility trail" gap called out in the README's Scope and limitations section: a UI click-through becomes a script you can paste into RStudio, check into version control, or drop into a methods section. In faceted mode each facet gets its own R button (same pattern as the existing per-facet TXT report). Plot reproduction is deliberately out of scope — the script contains statistical tests only and says so in its header comment, matching the tool's existing honest-scoping posture.
- **R-script export from the Power Analysis result card** — the computed-result panel in `power.html` now shows the same `⬇ R` chip as soon as a valid answer is displayed. Clicking it emits a short `pwr`-based R script that mirrors the current configuration exactly: the chosen test (`pwr::pwr.t.test` with `type = "two.sample" / "paired" / "one.sample"`, `pwr::pwr.anova.test`, `pwr::pwr.chisq.test` with the idiomatic uppercase `N`, or `pwr::pwr.r.test`), the selected `alternative = "one.sided" / "two.sided"`, and every parameter the user typed in, with the one the toolbox was solving for passed as `NULL` so `pwr` solves for the same unknown. A trailing comment records the value the JS solver reported (`# Toolbox reported: n = 64` or `# Toolbox reported: power = 71.2%`) so the R output can be eyeballed against the toolbox answer without opening the benchmark suite. Same motivation as the StatsTile export — turn a click-through into a reproducible, one-line-to-install script that can be checked into a methods section.
- **Dark mode** — full light/dark theme with a sun/moon toggle on the landing page and every tool's top bar. First visits to a tool follow `prefers-color-scheme`; the landing page defaults to light regardless of system preference so the gallery reads as a neutral entry point. Choices persist in `localStorage` and sync across the landing page and all open tool iframes. Plot cards stay white in both themes so exported SVG/PNG always render on a white canvas.
- **Component CSS classes** — introduces `tools/components.css` with `dv-panel`, `dv-input`, `dv-label`, `dv-select`, `dv-btn-*`, and the `dv-num` stepper. Adds `:hover`, `:focus-visible`, `:active`, and `:disabled` states that inline styles couldn't express.
- **−/+ numeric steppers** — all `<input type="number">` fields across the tools now render as a compact stepper with `−`/`+` buttons (click or press-and-hold to repeat) replacing the native browser arrows.
- **Bar outline controls in Group Plot (bar style) and Aequorin inset barplot** — unified "Bar outline" checkbox + width slider + color picker across both tools.
- **Named SVG group ids for Inkscape** — every chart export wraps its elements in `<g id="…">` groups with human-readable ids (`background`, `grid`, `axis-x`, `axis-y`, `data-points`, `plot-frame`, `legend`, `cld-annotations`, `significance-brackets`, …) plus per-series ids. Inkscape surfaces these in its Objects panel so users can select and edit grouped elements by name. The shared SVG legend renderer now wraps the whole block in `<g id="legend">` and gives each sub-block its own id — scatter emits `legend-color` / `legend-size` / `legend-shape`, and boxplot/aequorin derive ids from block titles so the individual pieces can be selected independently in Inkscape. `stats-summary` is also now a named group in boxplot exports.

- **Logarithmic value axis for Group Plot** — a "Y scale" dropdown in the Plot params panel lets users switch the value axis between Linear, Log₁₀, Log₂, and Ln (natural log). The transform is purely visual — statistics are still computed on raw data. Tick labels display original data values (1, 10, 100, not 0, 1, 2) following scientific convention. Log₁₀ shows sub-ticks at 2× and 5× multiples when the range spans ≤ 4 decades. Zero and negative values are clamped with a warning banner. Bar chart baseline moves to `yMin` in log space (no zero line exists). Works with all four plot styles, horizontal mode, facets, and subgroups. Annotation headroom (CLD letters, significance brackets) is computed in log space so labels don't overlap data.

- **Export filenames now carry the uploaded file's name** — dropping `analysis1.csv` into any tool produces exports like `analysis1_groupplot.svg`, `analysis1_scatter.csv`, `analysis1_venn_membership.csv`, etc. A new `fileBaseName(fileName, fallback)` helper in `shared.js` strips the extension and provides a clean fallback for pasted data (no file = tool name only, no leading underscore). Applied to Group Plot (SVG/PNG/sanitized CSVs + StatsTile TXT/R via `fileStem`), Aequorin (SVG/PNG, barplot, calibrated CSV, sums CSV, StatsTile), Venn (SVG/PNG, membership CSV, per-intersection CSV), and Scatter (cleanup: consistent fallback). Molarity and Power are excluded (no file upload). Faceted exports append the category after the base name: `analysis1_groupplot_Treatment.svg`.

- **`tinv` rewritten with closed forms + Newton-Raphson so extreme / heavy-tail quantiles match R to machine precision** — the original implementation was pure bisection with hard-coded `[-50, 50]` bounds, which silently clamped any quantile whose true value exceeded that range (worst case: the Cauchy df=1 tail, where `tinv(0.001, 1)` returned `-50` instead of `-318.3`). An earlier fix expanded the bisection bounds outward (same trick `chi2inv` uses) which unclamped the extremes but still only hit ~1e-4 absolute accuracy and could take hundreds of iterations deep in the tail. The current implementation is substantially stronger: `df = 1` uses the Cauchy closed form `-cot(π · p)`, `df = 2` uses `(2p − 1) / √(2p(1 − p))` (both exact), and `df ≥ 3` uses Newton-Raphson on `F(x) − p` seeded with a Cornish-Fisher correction to the normal quantile (`x = z + z(z² + 1)/(4·df)`), with a damped-step safety cap and a bisection fallback if Newton fails to converge. All branches fold upper-tail inputs (`p > 0.5`) into the left tail before solving so inputs like `p = 1 − 1e-12` don't suffer catastrophic cancellation. A small helper `tpdf` (t-PDF in log space, built from `gammaln`) is added alongside `tcdf` to provide the Newton derivative. Verified against R's `qt()` to ~1e-14 relative error across df ∈ {1, 2, 3, 5, 10, 15, 30, 100} and p ∈ {1e-15 … 1 − 1e-12}, including cases where the true quantile exceeds ±100 000. Covered by five tinv tests in `tests/stats.test.js` (standard critical values, closed-form df ≤ 2, extreme quantiles, upper-tail antisymmetry, and tcdf round-trip).
- **Statistical tests now refuse zero-variance input instead of silently emitting NaN/Infinity** — `tTest` (both Student and Welch branches), `tukeyHSD`, and `gamesHowell` previously divided by zero whenever the user fed them perfectly constant groups (e.g. a replicate set where every value was identical, or a control group with no measurable spread). In the Welch branch that produced a `NaN` Welch-Satterthwaite df, which propagated through `ptukey` into `NaN` p-values; in Student's branch it produced `±Infinity` t-values; in `tukeyHSD` it produced `Infinity` q-statistics from zero MSE. None of these cases were flagged to the user — the statistics tile would either render blank cells or, worse, report artefact p-values that could mislead downstream interpretation (and poison the compact-letter display, since `NaN >= alpha` is false and CLD would treat the pair as "significant"). Each function now detects the degenerate case upfront and returns a populated `error` field that `shared-stats-tile.js` already surfaces as `⚠ Data are essentially constant …`, matching R's `t.test` convention. Only the all-constant case is blocked — mixed inputs where a single group happens to be constant among varying ones still compute correctly (the Welch-Satterthwaite df degenerates cleanly to `n_other − 1`, and Tukey HSD's MSE is driven by the non-constant groups). Covered by seven new degenerate-input tests in `tests/stats.test.js`.
- **`hedgesG` uses the exact gamma-ratio bias correction + doc clarifications on `fFromGroupMeans` and `selectTest`** — `hedgesG` was multiplying Cohen's `d` by the asymptotic shortcut `J ≈ 1 − 3/(4n − 9)`, which is the leading term of an expansion that's off by ~0.27 % at the smallest practical sample (n1 = n2 = 3, df = 4: shortcut J = 0.8 vs. exact J = Γ(2)/(Γ(1.5)·√2) ≈ 0.7979). Now computes `J(df) = exp(gammaln(df/2) − gammaln((df−1)/2) − ½·log(df/2))` in log space via `gammaln` to avoid Γ overflow at large df, matching what R's `effectsize::hedges_g()` reports. Cosmetic at typical sample sizes but exact at the small-n end. Also adds two doc clarifications: (1) `fFromGroupMeans` now spells out that it uses the **population-style** SD of the group means (divides by k, not k−1) — the Cohen (1988) and `pwr::pwr.anova.test` convention the downstream power calculator is calibrated against. (2) `selectTest`'s docstring now explains the family-wise α inflation from per-group Shapiro-Wilk: with k = 5 groups at α = 0.05 the chance of falsely declaring at least one group "non-normal" is ~23 % even when all five are perfectly normal, biasing the auto-pick toward Kruskal-Wallis. The bias is by design (ANOVA is robust to modest non-normality and the auto-pick is an entry-point heuristic, not a publication-grade decision rule), but users who already know their data are normal can loosen `alphaNormality` (e.g. 0.01) or override the test pick directly. Two new tests in `tests/stats.test.js` pin the exact `J(4)` value and verify convergence to the asymptote at large n.
- **`sampleVariance` rewritten with Welford's one-pass algorithm and `chi2inv` rewritten with Newton-Raphson** — two related numerical-precision improvements in `stats.js`. (1) `sampleVariance` was using the standard two-pass algorithm: compute the mean in a first pass, then sum `(x[i] − mean)²` in a second pass. Numerically stable on most data, but it reads the array twice and is no more accurate than Welford on offset inputs. The Welford recurrence (`mean += (x[i] − mean) / (i + 1)`; `M2 += (x[i] − mean_old) · (x[i] − mean_new)`) is one pass, equally stable, and slightly faster on large arrays. Verified against the same `[2,4,4,4,5,5,7,9]` test case and against shifted variants at offsets up to 1e12 (where Welford holds 1e-3 precision while the input itself is starting to round at the IEEE 754 mantissa limit). (2) `chi2inv` was pure bisection with a doubling upper bound and a fixed `(hi − lo) < 1e-10` termination — correct, but ~50 iterations per call and inconsistent with the new `tinv` (which uses Newton with Cornish-Fisher seeding from commit dd060e4). It now uses Newton-Raphson seeded from the Wilson-Hilferty cubic-normal approximation `(X/k)^(1/3) ≈ N(1 − 2/(9k), 2/(9k))`, with strict guards (only accept iterations that stay positive and within `[x/2, 2x]`) so it can't overshoot in the saturated tails where the χ² PDF collapses below 1e-12. When Newton's guards trip — typically deep tails at small `k` — it falls through to the same doubling-and-bisect fallback as before, and a new `chi2pdf` helper is added alongside `chi2cdf` for the Newton derivative. Returns `NaN` for `k ≤ 0`. Verified against R's `qchisq()` to 1e-5 absolute on standard critical values and to 1e-8 round-trip self-consistency across `k ∈ {1, 2, 5, 10, 30, 100}` and `p ∈ {0.001 … 0.999}`. Covered by six new tests in `tests/stats.test.js` (sampleVariance shift-invariance under offsets, four `chi2inv` test groups for standard values, small df, round-trip, and degenerate inputs).
- **`compactLetterDisplay` and `kruskalWallis` no longer silently corrupt on degenerate input** — two related defensive guards in `stats.js`. (1) `compactLetterDisplay` walked its pair list with `if (p >= alpha) continue`, but `NaN >= alpha` is `false` in JavaScript, so any pair whose upstream test had failed (e.g. a Tukey/Games-Howell pair from a zero-variance group) was treated as _significant_ — the loop would split letters on noise and produce arbitrary garbage labels. The guard now checks `Number.isFinite(p)` first and skips NaN pairs as "no evidence of difference", matching `multcompView::multcompLetters` in R which requires non-NA inputs. (2) `kruskalWallis` had no explicit handling for the all-tied case (e.g. `[[5,5],[5,5],[5,5]]`): ranks were all `(N+1)/2`, `H` computed to a 0/0 form that landed on `H = 0`, the tie-correction denominator `C` was also 0, the `if (C > 0)` guard skipped the divide, and the function returned `{ H: 0, df, p: 1 }` — a "no significant difference detected" answer that masked the fact that the test was undefined. Now matches R's `kruskal.test` behavior (warns and returns NaN) by detecting the all-tied case explicitly and returning `{ H: NaN, df, p: NaN, error: "Data are essentially constant" }`, which `shared-stats-tile.js` already surfaces. Partial ties (mixed tied + non-tied data) continue to compute normally — only the fully degenerate case is rejected. Covered by five new tests in `tests/stats.test.js` (CLD all-NaN pairs, CLD mixed NaN + significant pairs, KW all-tied uniform groups, KW all-tied uneven groups, and a regression test confirming partial ties still compute).
- **Root-finder bracket expansion in `bisect`, `qtukey`, and `powerAnova`** — three places in `stats.js` were using fixed-width brackets that silently clamped to a stale endpoint when the true root lay outside, instead of failing or expanding. (1) `bisect` now refuses to run when the target is not bracketed by `[fn(lo), fn(hi)]` and returns `NaN` — historically it would simply walk to whichever endpoint was nearer, letting downstream code use a number that wasn't actually a root. All existing callers either expand the bracket first or pass conservative bounds, so the change is invisible in normal use but protects against future bracket bugs. (2) `qtukey` (inverse studentized range) was hard-coded to `bisect(…, 0.01, 100)`, which clamped silently for extreme inputs — e.g. `qtukey(0.999, 50, 1)` returned `~99.999` (the saturated upper endpoint) instead of expanding the bracket. It now doubles `hi` up to 20 times until it covers `p`, then bisects with relative-tolerance termination (`1e-7 * (|lo| + 1)`) so precision is consistent regardless of the answer's magnitude. Also rejects degenerate `(k < 2 || df < 1)` inputs upfront. (3) `powerAnova` similarly hard-coded its critical-F search to `bisect(…, 0, 200)`. The central F distribution at `(df1, df2) = (1, 1)` only reaches `p = 0.955` at `x = 200`, so any α ≤ 0.045 at that df pair silently clamped `fCrit`. The bracket is now expanded by doubling (cap 20 doublings → ~2·10⁸) until it covers `1 − α`. Covered by eight new tests in `tests/stats.test.js` (bisect bracket-failure cases, qtukey extreme-tail expansion, qtukey degenerate-input rejection, and powerAnova heavy-tail recovery).
- **`oneWayANOVA`, `welchANOVA`, `leveneTest`, and `cohenD`/`hedgesG` also refuse zero-variance input** — a follow-up audit found four more functions hitting the same zero-variance failure mode as `tTest`/`tukeyHSD`/`gamesHowell`, each silently returning a phantom result on constant data like `[[1,1,1],[2,2,2],[3,3,3]]`. `oneWayANOVA` had a pre-existing `ssWithin === 0` branch that returned `{ F: Infinity, p: 0 }` — a fabricated "p < 2.2e-16" significance that was a divide-by-zero artefact, not evidence of a real location shift (R's `aov` does the same thing with a warning, but that warning never reached the UI). `welchANOVA` computed weights as `n_i/s_i²` and silently went `Infinity → NaN` the moment any group had `s² = 0`. `leveneTest` had the same buggy `ssWithin === 0 → F = Infinity` path as `oneWayANOVA` but on the absolute-deviations ANOVA. And `cohenD` (plus `hedgesG`, which wraps it) divided the mean difference by a zero pooled SD and returned `±Infinity`. Each is now guarded: the three test functions return a populated `error` field (matching the `tTest` convention so `shared-stats-tile.js`'s `⚠ Data are essentially constant …` banner picks them up automatically), and `cohenD` returns `NaN`. `welchANOVA` refuses on _any_ zero-variance group (matching R's `oneway.test(var.equal = FALSE)` behaviour — the weighting scheme can't tolerate a single infinite weight), while `oneWayANOVA` and `leveneTest` only refuse the all-constant case since mixed inputs still have positive within-group dispersion. Covered by six new tests in `tests/stats.test.js`.

### Fixed

- Power tool: one-way ANOVA sample-size solver no longer jumps to 100000 for large effect sizes at k ≥ 6. Root cause was `ncf_sf` in `tools/stats.js` truncating the Poisson mixture at a fixed ±500-step window, which is narrower than σ = √(λ/2) once λ gets large (tens of thousands). Truncated sums returned bogus values around 0.7–0.8 at huge n, violating the monotone-in-n assumption `bisect` depends on and driving it toward `hi = 100000`. Fix widens the Poisson window to ±8σ with a `pUp < 1e-14` early-exit, and adds a normal-approximation short-circuit for the far tails (|z| > 6) when λ > 1000 and d₂ > 4, using the closed-form NCF mean / variance. All 217 existing power tests still pass unchanged.
- `gammainc` (regularized lower incomplete gamma P(a,x)) was silently returning wrong answers for large `a`. The series branch (taken when x ≤ a+1) has a fixed 200-iteration cap, but when x ≈ a the series terms form a Poisson-like bump of effective width √a — at a = 10000 it needs ~780 iterations to converge, so the loop was exiting mid-bump. Observable symptom: `gammainc(10000, 10000) = 0.478` (correct value ≈ 0.501), propagating to `chi2cdf(50000, 50000) = 0.397` (should be 0.5) and every large-df χ² / F tail computation that routes through it. Fix scales the iteration cap with √a on both the series and continued-fraction branches. Found by an audit sweep of stats primitives at extreme parameters after the `ncf_sf` bug.
- `ncchi2cdf` (noncentral χ² CDF) had the same truncated-Poisson-mixture pathology as `ncf_sf`: a fixed ±500-step window around the mode of the Poisson mixture, which is narrower than √(λ/2) once λ gets large. At λ = 100000 the CDF at 3× the mean returned 0.987 instead of 1. Fix widens the Poisson window with √(halfLam), adds a `pUp < 1e-14` early exit, and short-circuits far tails (|z| > 6) via normal approximation using the closed-form noncentral χ² mean (k+λ) and variance (2(k+2λ)).
