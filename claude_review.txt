# Code Review — dataviz

Context: solo-dev, lab-internal browser-only tool for plant
scientists. Target audience is students and occasional users who
need to go from pasted CSV to a publication-grade SVG without
installing anything. Not commercial, no SLA, no external users.
Review calibrated to that reality.

## Real bugs — fix

### `kde()` in `tools/shared.js` is O(n²)

```js
const bw = 1.06 * Math.min(
  Math.sqrt(
    values.reduce((s, v) => s + (v - values.reduce((a, b) => a + b, 0) / n) ** 2, 0) / n
  ),
  (iqr || 1) / 1.34
) * n ** -0.2 || 1;
```

The inner `values.reduce((a, b) => a + b, 0) / n` recomputes the
mean inside the outer reduce callback — one full pass per element.
For n=10k that's 100M ops; for n=50k the violin/raincloud plot
hangs for seconds on modest hardware. Precompute the mean once.

Behavior-preserving fix, math is identical.

## Things that look wrong but are right for your context

- **Vendored React in `/vendor/`** — no CDN dependency, works
  offline, reproducible. For a lab tool shipped via GitHub Pages
  this is exactly right. A commercial SPA would use a bundler and
  tree-shake; you shouldn't.
- **Plain-JS `shared.js` / `shared-components.js` via script
  tags** — breaks the modern-bundler rulebook but it's the
  simplest thing that works with per-tool iframes and no build
  coupling between shared code and tools. Documented in CLAUDE.md,
  intentional, keep it.
- **Custom test harness (`tests/harness.js`) instead of Vitest /
  Jest** — 217 tests, fast, zero dependencies, runs in plain
  Node. For a solo project this beats adopting a framework you'd
  rarely touch.
- **Committed compiled `tools/*.js`** — unusual for a Node
  project, correct for static GH Pages hosting. The alternative
  is a build-on-deploy CI step, which you also have — but keeping
  the artifacts committed means `file://` opens work too, which
  matters for lab machines behind firewalls.
- **No TypeScript on `shared.js`** — ambient `.d.ts` covers the
  consumption side. Converting `shared.js` itself would force a
  build step on a file that currently needs none. Not worth it.
- **No router, no state management library** — each tool is a
  self-contained App component. For 6 tools with zero
  cross-navigation, anything else would be ceremony.

## What's genuinely good

- **Iframe isolation** — one tool crashing can't kill the
  landing page, and each tool gets a clean React root. Simple,
  effective, and now paired with error boundaries.
- **Data flow is linear and debuggable** — paste → parse →
  assign roles → compute → render. Every step is a pure function
  you can test in isolation, and you do test them.
- **CSV parsing handles the real-world mess** — auto-separator
  detection, decimal-comma fix, wide↔long reshape. This is the
  part that would bite students hardest if it didn't "just work,"
  and it does.
- **Export paths (SVG/PNG/CSV) on every tool** — this is the
  actual deliverable for the target audience (figures that go in
  reports / theses). Good that it's uniform.
- **CI is proportionate** — lint + format + typecheck + tests +
  build. Not overbuilt, not underbuilt. The Node 22/24 opt-in
  shows you're tracking upstream changes without chasing them.
- **Error boundaries + version-from-git-describe** — recent
  additions that show the project has matured past "it works on
  my machine."
- **CLAUDE.md** — genuinely useful for onboarding (even if the
  only person being onboarded is future-you or an AI assistant).

## What I'd invest in (not urgent)

- **A single screenshot per tool in the README** or landing
  page tooltip. Students decide whether a tool fits their data in
  ~3 seconds of looking. Text descriptions lose to pictures.
- **A "load example data" button on each tool.** Lowers the
  barrier for the "I just want to see what it does" visitor from
  "find a CSV" to "click once."
- **Consider extracting `kde()`, `quartiles()`, `computeStats()`
  into `tools/stats.js`** — only if `shared.js` grows past ~800
  lines. Not now.
