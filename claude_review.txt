# Code Review — dataviz
# Revision 2 — 2026-04-13

Context: browser-only data visualization toolbox for plant scientists.
Originally lab-internal; now potentially published on Zenodo for broader
scientific use and citation. Review calibrated to that wider audience.

## What was fixed since the last review
────────────────────────────────────────────────────────────────

### kde() O(n²) bug — FIXED

The inner `.reduce((a,b) => a+b) / n` inside the outer reduce is gone.
Mean and variance are now precomputed once before the bandwidth
calculation. Confirmed in `tools/shared.js:494–496`.

### 3-digit hex color — FIXED

`normalizeHexColor()` in `shared-components.js:6–14` now expands `#abc`
to `#aabbcc`. Both forms are accepted and round-tripped correctly.

### Whisker clamping edge case — FIXED

Interpolated Q3 could fall between two data points, making `wHi < q3`
when the larger neighbor was an outlier. Clamped:

  wLo: Math.min(s.find(v >= q1-1.5*iqr) ?? s[0], q1)
  wHi: Math.max([...s].reverse().find(v <= q3+1.5*iqr) ?? s[n-1], q3)

Outlier dots still render at their actual values; only the whisker
endpoints are clamped. Correct behavior, correct fix.

### var → const/let — DONE throughout

## Real bugs — fix
────────────────────────────────────────────────────────────────

### `shared-components.js` is 2711 lines and growing

Not a bug yet, but a maintenance risk. The file now contains inputs,
layout, banners, the long-format pipeline (ColumnRoleEditor,
FilterCheckboxPanel, RenameReorderPanel, GroupColorEditor), SVG legend
rendering, StatsTile (the largest component), and ErrorBoundary.

When a single file needs to change for 5 unrelated reasons in one
session, it creates unnecessary merge risk and makes code review harder.
At this size, a split into two files would be justified:
  - `shared-components.js`   — generic inputs/layout/banners
  - `shared-groupplot.js`    — long-format pipeline + StatsTile

The constraint (plain-JS, no import/export, loaded as `<script>`) still
holds for both — you'd just add a second `<script>` tag in the tools
that need it.

Not urgent, but worth doing before the file reaches 3500+ lines.

### scatter.tsx and venn.tsx have no "load example data" button

boxplot.tsx and aequorin.tsx both have `loadExample` callbacks and
`makeExamplePlantCSV()` / example TSV data. The other four tools force
the user to supply a file to see anything — a friction point for new
users or reviewers evaluating the tool for the first time.

Power and Molarity don't accept file input (they're calculators), so
they're fine. Scatter and Venn are the gap.

## Things that look wrong but are right for this project
────────────────────────────────────────────────────────────────

All points from the previous review still hold:

- **Vendored React in `/vendor/`** — correct for offline/lab/GH Pages.
- **Plain-JS shared files via `<script>` tags** — intentional, documented
  in CLAUDE.md, the right tradeoff for this architecture.
- **Custom test harness** — 485 tests, zero framework dependencies,
  fast. No reason to change.
- **Committed compiled `tools/*.js`** — correct for static hosting;
  `file://` works locally without a build step.
- **TypeScript strict: false on `.tsx` files** — light typing only.
  Correct for a project where the runtime constraint is no bundler on
  shared code. The ambient `.d.ts` covers consumption.
- **No router, no state management** — right for 6 self-contained tools.

## What's genuinely good (updated)
────────────────────────────────────────────────────────────────

### Statistical layer is now seriously solid

`stats.js` (1323 lines) implements distributions (normal, t, F, χ²,
non-central variants), six statistical tests, five effect size measures,
three post-hoc procedures, Shapiro-Wilk normality, Levene's homogeneity,
and automatic test selection with decision logic that mirrors what a
statistician would actually do. 99 dedicated tests + a full R 4.5
cross-validation benchmark (285 comparisons) give strong confidence in
numerical correctness. For a scientific tool where users trust the p-values,
this matters a lot.

### Test coverage is now proportionate to the code

485 tests across 6 suites, all passing. The test surface now matches
the code surface: statistics, power analysis, parsing, components, and
integration paths are all covered. The R benchmark catches floating-point
divergences that unit tests would miss.

### Privacy story is a genuine scientific strength

Zero backend, zero tracking, zero data upload — all processing is in the
browser. For life-science labs handling unpublished data or working under
data governance policies (GDPR, institutional IRB), this is a real
differentiator worth stating clearly in the README.

### Export quality

SVG + 2× PNG on every chart tool. SVG is publication-ready vector.
Consistent, tested path.

### Iframe isolation + ErrorBoundary

One tool crashing cannot affect another. ErrorBoundary catches React
render errors within a tool. Robust for an environment where users paste
arbitrary data.

### Accessibility

All SVG chart roots have `role="img"`, `aria-label`, `<title>`, `<desc>`.
Data-bearing groups have `aria-label`. Decorative SVGs are
`aria-hidden`. Solid foundation, especially important for a broader
scientific audience.

## What to invest in — now that Zenodo publication is on the table
────────────────────────────────────────────────────────────────

### 1. README needs a rewrite before publication

The current README opens with:
  > "An entirely vibe-coded application with claude"

This is charming for a lab-internal tool and will be a credibility problem
the moment the DOI is cited in a paper. Suggested framing:

- Lead with what it does and who it's for.
- State the privacy guarantee prominently.
- Add a "How to cite" section with the Zenodo DOI and a suggested
  BibTeX entry.
- Remove the vibe-coding line or move it to an Acknowledgements section.

The tool table and feature table are already well-structured — keep them.

### 2. Add a LICENSE file if you don't have one

Zenodo requires a license declaration. Without a clear license, downstream
users technically cannot legally use the code. MIT or Apache 2.0 are the
standard choices for open science tools.

### 3. Consider a CITATION.cff

GitHub and Zenodo both recognize `CITATION.cff` at the repo root and
surface it in the "Cite this repository" button. Costs 10 lines, saves
every user from having to format the citation manually.

### 4. Screenshots in the README (still the highest-ROI improvement)

One screenshot per tool. Scientists decide whether a tool fits their data
in ~3 seconds. The README's text descriptions are good but lose to a
picture. For a Zenodo publication this also helps reviewers and editors
understand the tool without running it.

### 5. Example data on scatter and venn

See "Real bugs" above. This is especially important for a publicly
accessible tool where many visitors arrive to evaluate, not to work.

## Overall score, recalibrated for potential Zenodo publication
────────────────────────────────────────────────────────────────

**9.0 / 10** for a scientific open-source browser tool.

The codebase is in genuinely good shape. The statistical engine is
carefully implemented and cross-validated against R. The test suite is
proportionate. The architecture is well-reasoned and documented. The
privacy model is strong for scientific use.

The gap to 10/10 is almost entirely presentation and meta-data: the
README frames this as a joke project, there's no license declaration,
no citation format, and no screenshots. Those are one-afternoon fixes
that matter for the difference between "lab curiosity" and "citable
scientific software."

The one real code issue is `shared-components.js` heading toward
unmanageable size — but it's not there yet.
