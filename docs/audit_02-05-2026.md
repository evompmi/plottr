## Tier 1 — bugs and quiet failures (worth fixing soon)

### #1 — Exported R scripts still branded "Dataviz Toolbox", not "Plöttr"

The 1.0.4 release renamed the project from "dataviz" / "Toolbox" to "Plöttr"
across UI chrome and the benchmark page. Six R-export functions in source
**were missed** and still emit `# Dataviz Toolbox — …` as the first line of
every R script users download:

- `tools/shared-r-export.js:105` — generic header
- `tools/shared-r-export.js:348` — power-analysis header
- `tools/lineplot.tsx:1127` — Line Plot R script export
- `tools/aequorin/reports.ts:197` — RLU timecourse R script export
- `tools/boxplot/reports.ts:99` — Group Plot R script export
- `tools/heatmap/reports.ts:62` — Heatmap R script export

User impact is high: the script header is the first thing a user sees when
they open the downloaded `.R` file in RStudio, and these files often get
archived alongside published figures. Same drift class the 1.0.4 fix flagged
in the benchmark generator — fix everywhere, not just the spot you noticed.
Also worth a one-line `tests/r-export.test.js` regression assertion that the
header literal is `"Plöttr"` so a future rename doesn't have to grep again.

### #2 — `scripts/write-version.js` `fromChangelog()` regex has no test footprint

The race fix in `4863413` made the deploy pipeline rely on a regex pulling
the version from the first concrete `## [X.Y.Z] - DATE` heading in
`CHANGELOG.md`. The regex (`/^## \[(\d+\.\d+\.\d+)\]/m`) is small but it's the
single point of failure for every future Pages deploy showing the right
version. A future change to CHANGELOG formatting (e.g. someone tries
`## [1.0.6] (2026-06-15)` with parens, or adds a pre-release tag like
`## [1.1.0-beta]`) silently falls through to the `git describe` fallback,
which on the deploy runner is the last reachable tag — re-introducing the
race we just fixed.

Fix: add `tests/write-version.test.js` covering (a) typical case, (b)
`Unreleased` skipped, (c) malformed CHANGELOG → `null`, (d) no concrete
release yet → `null`. Pattern to copy: `tests/handoff.test.js` (vm context +
in-memory fs stub, ~40 lines).

## Tier 2 — drift and inconsistency

### #3 — `postMessage` targets always use `"*"`; receivers don't check `e.origin`

Project-wide pattern, four senders + two receivers:

- Senders (`postMessage(payload, "*")`): `index.html:1134`, `venn.tsx:441,445`,
  `aequorin/plot-area.tsx:435`. The `"*"` target lets any iframe inside a
  malicious embedder read the message contents (tool switch commands, theme
  flips, hand-off payloads). Contents are non-secret — the leakage is benign
  — but the hygiene is below the bar set by `upset.tsx:2622`, which already
  pins `window.location.origin` for the venn → upset hand-off.
- Receivers without origin check: `index.html:1048` (the `openTool` channel
  every cross-tool nav uses, including the new `Open in Boxplot` button I
  just shipped), `theme.js:145`. A malicious page embedding Plöttr in an
  iframe can spam `{type: "openTool", …}` to switch tabs or `{type:
"dataviz-theme-set", theme: "dark"}` to flip themes. Worst-case impact: UI
  thrashing, no data exfil.

Trivial fix:

- All four senders → `postMessage(payload, window.location.origin)`.
- Both receivers → `if (e.origin !== window.location.origin) return;` as the
  first line of the handler. Mirrors `upset.tsx:2622`.

Single commit, ~8 lines net change. Doesn't violate the simple-spirit rule —
the app already has the pattern in one place, this just propagates it.

### #4 — `dv-tile-title` migration left inline-margin overrides at every call site

The `f9b2c34` unification correctly extracted typography into
`.dv-tile-title`, but the inline `style={{ margin: …, … }}` at each call site
still carries different margin values (`"0 0 8px"`, `"0 0 6px"`, `"0 0 4px"`,
`"0"`). That's deliberate — different tiles have different bottom-spacing
requirements based on what's underneath them — but the divergence means
future contributors copying a call site need to think about which margin to
use.

Honest take: leave alone. The margins are layout, not typography, and the
class did its job (typography is now uniform). Adding a "default margin"
to the class would over-step into layout territory and break call sites
that genuinely need a different value. **No-op finding** — flagged because
the prior agent will mention it; don't act.

## Tier 3 — nice to have (skip unless free)

### #5 — Pre-commit hook stash-pop conflict surfaced today (recovered manually)

Today's `feat(aequorin): "Open in Boxplot" hand-off` commit hit the
hook's documented failure mode: the stash-pop conflicted because the
sourcemap I had unstaged drifted from the rebuilt sourcemap. The hook
correctly preserved the stash (per the audit-23 #14 design — "leave the
stash in place with a clear warning") and exited 1, but the recovery
required `git stash drop` + manual `git add` of the freshly rebuilt
file.

The current hook output already explains what happened ("WARNING:
git stash pop conflicted — your unstaged work is preserved in
stash@{0}"). Worth adding two lines to that warning showing the exact
recovery command for the common case ("if the conflict is a
re-built compiled output — `git checkout --theirs <file>` and then
`git stash drop`"). But this is polish on a low-frequency event;
defer indefinitely.

### #6 — Test counter procedure has CI safety net but no auto-bump

`index.html` line 554 + 815 carries `907 internal tests`. CI verifies it
matches the `npm test` total (`test.yml:42-51`), so drift can't ship — but
the bump itself is a manual step contributors need to remember in the same
commit as the test addition. Auto-bumping (e.g. via a `prebuild` step that
runs `npm test --silent` and rewrites `index.html`) would (a) require
running tests on every build, which slows the watch loop, and (b) couple
build to test execution. Current setup (CI catches the drift, contributor
fixes once, lesson sticks) is fine. **Defer.**

new infrastructure. The recent additions (`shared-handoff.js`,

shape; the audit confirms the shape is consistent. CHANGELOG, CI badge
