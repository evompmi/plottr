Equivalent mutants — Stryker generates _syntactically-different_ but
_semantically-identical_ code. No test can kill them by definition.
Common pattern: `if (n < 2) return …` mutated to `if (false) return …`
where another check downstream catches `n < 2` anyway, so the redundant
guard is invisible to any test.

### Note on the stats.js result

**First pass — file-level skip (14.87% raw / 55.37% covered, 521 killed,
2563 no-coverage).** `tests/stats.test.js` was no-op'd entirely under
Stryker because a handful of its R-cross-validation tests run inner
numerical loops (qtukey at df=1, multisetIntersectionPExact at k=5/2
deep tails) that bloat ~3000× under Stryker's perTest-coverage probes,
overrunning any reasonable per-test budget. The whole-file skip dropped
~209 tests from coverage and pushed 73% of mutants into no-coverage —
the score reflected what `stats.property.test.js` (49 properties) could
reach on its own.

**Second pass — selective skip (57.72% raw / 68.02% covered, 1946 +
68 timed out / 947 survived / 528 no-coverage).** Replaced the file-
level skip with a `SKIP_UNDER_STRYKER` Set of 3 test names: the qtukey
df=1 bracket-expansion test and the two pathological cpsets cases.
`vitest.config.js` was also taught to bump `testTimeout` to 300 s _only_
when Stryker is running (cwd contains `.stryker-tmp`); regular
`npm test` keeps the 60 s ceiling. 206 of 209 R-cross-validation tests
now run under Stryker. Wall clock for the mutant phase grew from
~14 min to ~30 min, which is fine since Stryker isn't a CI gate.

This is **path 3** of the three paths sketched after the first pass —
shard the slow tests by name rather than skipping the file or the whole
suite. It worked: the raw score nearly quadrupled and the no-coverage
pool shrank from 2563 → 528.

What's left at 947 survivors:

- **Precision-shifting on A&S polynomial coefficients.** Common pattern:
  `c1 * lnN * lnN * lnN` mutated to `c1 / lnN * lnN * lnN` — the
  remaining `lnN * lnN` algebraically simplifies, and a small
  coefficient flip past R's quoted precision (~1e-9) is below what
  cross-validation can resolve.
- **Equivalent mutants** in defensive guards that another check
  downstream catches (the same pattern observed on scatter / lineplot).
- **Functions with no test coverage at all** — hclust, kmeans,
  dendrogramLayout, leveneTest, parts of multisetIntersection. These
  are responsible for the 528 no-coverage residual.

Two further paths if pushing higher matters:

- Add property tests for the uncovered functions (hclust / kmeans /
  dendrogramLayout / leveneTest). Would chip away at the no-coverage
  pool and likely lift the raw score a few more points.
- Try `coverageAnalysis: "all"` instead of `"perTest"` — single-pass
  instrumentation has lower per-line overhead, possibly making the
  3 currently-skipped tests viable. Gain is uncertain; the 3 tests
  pin only ~10 lines of stats.js so the upside is bounded.

## Wins

- **Real test gaps caught and fixed.** Scatter's `return { valid: false }`
  → `return {}` slipped past every existing assertion (they all gated
  on `if (!reg.valid)`, which accepts `undefined` as falsy); ditto
  lineplot's silent ci95-zeroing at n ≥ 2. The fix in both cases was
  tightening the assertion to strict equality.
- **Loader-pattern problem identified and fixed for two tools.** Most
  other tools will need the same fix and the recipe now exists.
- **The fuzz → property migration was clean.** Every fuzz invariant
  has a corresponding property in the per-tool property file, plus
  extras (idempotency, boundary cases, sign-symmetry) that the
  imperative loop couldn't easily express. Coverage runs on every
  `npm test` instead of being gated behind `npm run fuzz:*`.

## Things I did sloppily

- **Weakened tolerance globally instead of bounding inputs.** First
  round of `tests/stats.property.test.js` loosened the `norminv`
  round-trip tolerance from 1e-6 to 1e-3 to make all values pass.
  The right move was to split the property: tight tolerance (5e-5)
  for the bulk |z| ≤ 3, looser (1e-3) for the tail, with a comment
  explaining the cdf-error amplification. Caught and refactored after
  pushback.
- **Left zombie Monitor processes parked on `tail -f`.** Each Stryker
  run armed a fresh Monitor; I forgot to `TaskStop` them after the
  completion event fired. They sat alive until the 60-min timeout
  hit. Pattern correction: stop each Monitor as soon as its
  completion event arrives.
- **Volcano's 100% mutation score is partly luck.** Volcano helpers
  are inlined into the compiled SPA bundle (`tools/_app/index.js`),
  which the render-smoke tests load through a path Stryker _can_ see.
  Other tools whose helpers aren't reached by the render path scored
  0% under the same configuration until the loader was refactored.
  Earlier write-ups didn't flag this clearly.
