# Pl√∂ttr v1.0.1 ‚Äî release notes

_Released 2026-04-23._

A long polish-and-fix release: per-group colour choices now round-trip
through Save / Load prefs, the AesBox header bands desaturate, the tail-
accurate stats path lands (`normsf`, `_wprob_upper`, `ptukey_upper` etc.) so
deep-tail p-values stop underflowing, the SuperExactTest-style multi-set
intersection significance plumbing lands across `stats.js` + UpSet, fuzz
harnesses get pulled into CI, and 30-odd UI / consistency / contrast fixes
sweep across the tool suite.

## Fixed

**Group / category colour choices now round-trip through Save / Load prefs and the auto-persist localStorage slot.** The PrefsPanel Save-to-file + Load-from-file flow and the debounced auto-persist only captured what's in `vis` (the reducer state driven by sidebar controls); per-group colours lived OUTSIDE `vis` as local `useState` in each tool, so colour picks silently dropped on reload. Moved the colour maps INTO `vis` ‚Äî lineplot's `groupColors`, boxplot's `boxplotColors` and `categoryColors` ‚Äî so the existing prefs machinery covers them with no changes to `shared-prefs.js`. Exposed as read-through `const name = vis.name || {}` plus a `useCallback` setter that accepts both direct-value and functional-updater calls (preserving every existing call site). New keys default to `{}` in `VIS_INIT_LINEPLOT` / `VIS_INIT_BOXPLOT`. On reload, names that match a saved entry pick up the stored colour; mismatched names fall back to the palette ‚Äî so switching datasets is non-destructive.

## Changed

**AesBox header bands desaturated ‚Äî pale tint + deep text instead of saturated fill + white text.** The earlier headers (slate `#475569` / emerald `#16a34a` / purple `#9333ea` with white text in light mode) read as "first-draft dashboard" / Stabilo-highlighter next to the rest of the understated UI. Swapped to low-saturation tints with the label text in the deeper role hue: slate-200 (`#e2e8f0`) behind slate-700 text; green-100 (`#dcfce7`) behind green-700 text; violet-100 (`#ede9fe`) behind violet-700 text. Dark-mode counterparts match the "dark ‚â§ light brightness" convention (slate-700 behind slate-400; dark green tint behind green-300; dark purple tint behind violet-300). New CSS var pair `--aes-*-header` (band tint) + `--aes-*-header-text` (label hue) per role in both light + dark blocks of `theme.css`. The three components that consume AesBox (scatter.tsx, lineplot.tsx, boxplot/steps.tsx) pick up the text colour through their `aesTheme` map instead of the previous hard-coded `"#fff"`.

**Boxplot Filter step ‚Äî live delta in the preview title + a 300 ms flash on the preview card when the kept-row count changes.** Filters whose effect falls past the first 10 preview rows used to give no visual feedback ‚Äî the title said `Preview (856 rows):` and the visible rows didn't change. Two compatible cues added. (1) The title now reads `Preview ¬∑ 856 of 1024 rows ¬∑ 168 filtered out` (the `‚Ä¶ filtered out` suffix hides itself when nothing is filtered so the default state stays clean). (2) Whenever `filteredRows.length` changes, the preview card remounts under a new React key, re-triggering a 300 ms CSS keyframe animation (`bp-filter-flash`) that flashes the background from `var(--warning-bg)` back to the default `var(--success-bg)` ‚Äî theme-aware, un-intrusive, and cheap. Standard React idiom for one-shot CSS animations (change a key to force remount, animation restarts). `DataPreview` is props-driven so the remount has no behavioural cost. The `FilterCheckboxPanel` walked every non-ignored column and, for the numeric value column, rendered a placeholder tile that read `"numeric ‚Äî use axis range in plot"`. The tile offered no affordance (value columns are never filtered away row-by-row ‚Äî rows are kept if the value parses) and gave users a misleading impression they could still re-pick which column was the value here. Added `colRoles[i] === "value"` to the early-return skip in `tools/shared-long-format.js`'s `FilterCheckboxPanel`. Group + filter columns still render their check-box tiles as before.

**Boxplot Configure step ‚Äî full-per-column `ColumnRoleEditor` replaced with a compact "Other columns" panel.** Now that `group` + `value` are assigned by the AesBox cards at the top, the editor below just needed to answer one question for each remaining column: "use as filter, or ignore?" The bulky per-column role dropdown (group / value / filter / ignore for every row) duplicated the AesBox cards' affordance and let users accidentally reassign the primary roles from the bottom of the screen. The shared `ColumnRoleEditor` helper stays in `shared-long-format.js` for backward compatibility but is no longer consumed anywhere.

**Boxplot column roles collapsed from five to four ‚Äî dropped the "text" role.** The Configure step previously offered `group / value / filter / text / ignore`. In practice `filter` and `text` were near-synonyms (both acted as aesthetic-mapping candidates for color / facet / subgroup in the plot sidebar), with the only difference being that `filter` columns also fed the Filter-step row-subsetter. Since row-filtering is opt-in there anyway, the distinction was a mental-model tax with no downstream payoff. Simplified to four roles: `group` (primary X), `value` (primary Y), `filter` (any number ‚Äî eligible for aesthetic mapping AND Filter-step subsetting), `ignore` (pretend this column doesn't exist). Touches: `tools/shared.js` (`roleColors` loses the text entry; `guessColumnType` returns `"filter"` instead of `"text"` for high-cardinality string columns), `tools/shared-long-format.js` (`ColumnRoleEditor` dropdown loses the text option; `FilterCheckboxPanel` numeric-column gate drops the `!== "text"` arm), `tools/boxplot/index.tsx` (`colorByCandidates` no longer includes text-tagged columns), `types/globals.d.ts` (`ColumnRole` type loses `"text"`, `guessColumnType` return type widens to the full `ColumnRole`), plus a test expectation update in `tests/parsing.test.js`.

**Heatmap ‚Äî cell-value tooltip now tracks the cursor correctly in dark mode and keeps working when the zoomed detail plot is open.** Two symptoms of one root cause: the tooltip used `position: fixed` with `clientX / clientY` (viewport coords), but `.dv-plot-card` gets `filter: brightness(0.85)` in dark mode (tools/theme.css:283), and a non-`none` CSS `filter` establishes a _new containing block for `position: fixed` descendants_ (per the CSS Transforms / Containment spec). That re-rooted the tooltip to the plot card instead of the viewport, so on dark-mode it appeared offset by the card's origin ("far away from the mouse"), and when the zoomed detail view opened the page stretched so the main card's origin shifted far enough that `cardOffset + clientXY` landed off-screen on both plots ‚Äî the tooltips "stopped working." Immune to any ancestor filter / transform / will-change ‚Äî the tooltip is positioned relative to a real `relative` ancestor that's always in the right place, in both light and dark mode, and on both main and zoom plots simultaneously.

**UpSet ‚Äî plot significance markers + bar coloring are now two-sided; depletion is no longer invisible.** Previously `sig.pAdj` was aliased to `pAdjUpper` (enrichment only), so the stars / `p=‚Ä¶` markers above each bar and the `pAdj < 0.05` cutoff for bar coloring only ever asked _"was this bar unusually tall?"_ A bar with `xObs=0` against `E[x]=5` ‚Äî a _significantly depleted_ intersection, a real "these sets avoid each other" finding ‚Äî would have `pUpper ‚âà 1` and render "ns" on the plot, silently discarding half the signal. Fix: each cached entry now also stores `pTwoSided = min(1, 2¬∑min(pUpper, pLower))` (textbook double-the-smaller-tail convention for a one-parameter Binomial) and a third BH family `pAdjTwoSided` is computed across every bar. `e.pAdj` (read by both the marker renderer and the bar-color rule) now holds the two-sided adjusted value. Stars / `p=‚Ä¶` / bar color all agree on one honest significance cutoff; direction (enriched vs depleted) is determined from `sign(observed ‚àí expected)` and drives the green/red coloring. The per-tail `pAdjUpper` / `pAdjLower` are still cached and shown in the ItemList panel's directional breakdown so analysts who want to see "how significant was the enrichment side specifically" still can. Marker renderer already consumed `sig.pAdj`, so no changes to `pStars` / `formatP`; just a wiring change in the batch-compute step.

**UpSet ‚Äî intersection-size labels now rotate -90¬∞ and the bar area reserves headroom for stacked rotated labels so they don't clip above the plot.** Horizontal size labels collided on dense plots (20+ bars with multi-digit sizes); the fix rotates them bottom-to-top with the same anchor math already used for the column ids and significance markers, which removes all horizontal overlap regardless of column width. The second half of the fix addresses a render-zone overflow: with both the size label _and_ a stars / p-value marker rotated and stacked above every bar, the combined vertical stack was taller than the gap between the tallest bar and `topPanelY`, spilling into (or above) the title area and getting clipped by the viewBox. The chart now computes a dynamic `labelHeadroom` from `topAxisMax` digits plus the widest possible significance-marker token ("\*\*\*\*" or "p=1.2e-99") at the active font size, and scales bars into `TOP_PANEL_H - labelHeadroom`. The axis line is re-anchored to `topPanelBottom - barAreaHeight` so it still matches the tallest tick. Per-bar sig-marker offset now uses that bar's actual size-string width (not a constant `fSize`), so short-size bars don't leave pointless gaps below their sig marker.

**UpSet ‚Äî intersection-statistics BH family no longer depends on display filters.** Previously `computeAllIntersectionStats` iterated over `truncatedIntersections` (the filter-visible subset), so the BH multiple-testing family was literally shifting with the `minSize` / `minDegree` / `maxDegree` sliders: raising the minimum size from 0 to 10 silently dropped small bars from the family, making the surviving bars' `pAdj` values look more significant than they were under the analyst's actual null. A user who set a filter then "computed stats" then later cleared the filter would see new bars appear without p-values while the old bars kept their now-inconsistent adjusted p-values (normalised over a smaller family). The fix tests every intersection under the active set selection ‚Äî the selection itself defines the null (which sets are in the scope of the hypothesis), but plot-side view filters are now purely visual. BH adjustment is computed once over the full family and cached; display filters decide which cached results to render. Button text updated to reflect the new semantics (`Compute stats (N intersections)` instead of `(N visible)`); sidebar hint note explicitly states that display filters do not change the BH family. Set-selection scoping is unchanged and intentional ‚Äî different scientific question, different set world, different null.

**`fixDecimalCommas` rewritten as per-column detection ‚Äî European thousand separators and US thousand separators no longer collide.** The previous implementation did a single global `/(\d),(\d)/g` rewrite across the whole text whenever the detected separator wasn't a comma. This silently destroyed US-format thousand separators (`1,000.50` ‚Üí `1.000.50` ‚Üí NaN, and the row was dropped), mutated label cells containing commas (`"E,coli"` ‚Üí `E.coli`, silent group rename), and touched quoted regions it had no business touching. The new implementation tokenises the text first, then decides per-column whether commas are decimal separators by counting how many cells become numeric with vs. without the rewrite AND requiring at least one cell in the column to have a non-3-digit fractional run (a pure `1,000` / `2,500` column looks like thousand-grouping and is left alone). Only flagged columns get rewritten; label columns and mixed-format columns are preserved exactly. Added 7 per-column detection tests and updated 3 pre-existing tests that pinned the old text-rewrite behaviour.
