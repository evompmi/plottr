# Plöttr v1.4.0 — release notes

_Released 2026-05-09._

This release is a "look-back-and-fix" cycle aimed squarely at the codebase
itself — hence the **Heavy refactor** label. The headline isn't a new tool or
a methodological shift; it's the architectural cleanup that follows two minor
releases of feature work. The plain-JS shared bundle was the largest source
of technical debt in the repo and has been substantially shrunk. Every plot
tool's React surface now type-checks end to end with zero `: any`. The 2566-
line `tools/stats.js` has been carved into five focused files. Volcano label
placement got a real algorithmic upgrade. The stats-test dispatch tables
have been consolidated behind one registry. Property-based testing expanded
to cover `stats.js` itself, and Stryker mutation testing is wired in as an
on-demand quality gate.

The prose below is the long-form context — what changed, why, and how — that
the in-cycle CHANGELOG entries grew to hold. The CHANGELOG itself now keeps
one bullet per item with a link back here so the file stays scannable.

---

## 🏗 shared→_shell migration — the typed scaffold

The repo started life as a single static-HTML page per tool, with shared
helpers concatenated from `tools/shared*.js` plain-JS files into
`tools/shared.bundle.js` and re-exposed as globals. That worked for the
iframe shell but was the largest source of technical debt left over from
v1.3.0's SPA migration: nineteen `tools/shared-*.js` files, all script-mode,
all mutually dependent through global names, no types, no module boundaries.

Across this cycle the bundle was migrated piece-by-piece into `tools/_shell/`
TypeScript modules with proper imports / exports / types:

- **Five individual migrations** moved `shared-prefs.js`, `shared-handoff.js`,
  `shared-discrete-palette.js`, `shared-svg-legend.js`, and `shared-core.js`
  into their typed `_shell/` siblings.
- **Cluster A** migrated `shared-color-input.js` and the long-format detector
  to `_shell/color-input.tsx` and `_shell/long-format.tsx`.
- **Cluster B** migrated `shared-file-drop.js` and the rest of `shared-ui.js`
  into `_shell/file-drop.tsx` and `_shell/ui.tsx`.
- **Cluster C** moved the largest pieces — `shared-stats-registry.js`,
  `shared-r-export.js`, and `shared-stats-tile.js` — into `_shell/`. This
  exposed `assignBracketLevels` and `computePowerFromData` as their own
  standalone `_shell/bracket-levels.ts` and `_shell/power-from-data.ts`
  modules during the cleanup.

End state: the plain-JS `shared.bundle.js` shrank from 19 → 7 files. The
remaining seven are intentional — `theme.js` (browser-side, sets CSS vars
before React mounts), `shared.js` (the parsing-and-color core consumed as
globals everywhere), and the five `stats-*.js` files (load order matters
across them, see below). They're the static-HTML pages' contract; folding
them into the typed scaffold would force a build-step on those pages.

### \_shell/ split + barrel

Five of the migrated `_shell/*.tsx` files had grown to host two or three
unrelated components each (`ui.tsx`, `long-format.tsx`, `core.tsx`,
`stats-tile.tsx`, `chart-annotations.tsx`). Each was split into per-component
files and re-exported through a new `_shell/index.ts` barrel. Naming
codified during the split: PascalCase.tsx for components, kebab-case.ts for
pure helpers, camelCase.ts for hooks. With `"sideEffects": false` in
`package.json`, esbuild tree-shakes per-symbol consumption through the
barrel — every tool that imports a single `_shell/` symbol pays only for
that symbol.

### Test-loader plumbing extraction

Each of the migrations added `esbuild.buildSync` + `vm.createContext` +
React-stub boilerplate to a different per-domain loader (`render-loader`,
`r-export-loader`, `lineplot-loader`, `boxplot-loader`,
`stats-dispatch-loader`, `discrete-palette-loader`, `prefs-loader`,
`handoff-loader`, plus the per-tool helpers loaders). The top-level shape
was ~80 % identical across all of them — only the modules to bundle and the
globals to expose differed. New `tests/helpers/_shell-test-utils.js` exposes
the small composable pieces (`builtins()`, `makeDomStubs()`,
`makeLocalStorage()`, `MINIMAL_REACT`, `bundleShell()`, `runCjs()`,
`requireViaTmpFile()`, `readSharedBundleSrc()`); seventeen loaders thread
through it, dropping ~600 lines of repeated plumbing. `render-loader` (real
React + happy-dom on `globalThis`) and `power-loader` (IIFE + symbol-export
splice for `power-app.tsx`) stay inline by design — their patterns are
different enough that folding them into the helper would muddy it.

## 🧬 React-tier typing campaign — every plot tool typed end-to-end

Through v1.2.0 / v1.3.0 the React surface of each plot tool was the loudest
source of `: any` annotations. The pure-helpers layer (`<tool>/helpers.ts`)
was already strict, but the `app.tsx` / `chart.tsx` / `controls.tsx` /
`steps.tsx` / `plot-area.tsx` files reached for `: any` whenever a
prop-bag or reducer-state shape grew unwieldy. This cycle closes that gap —
every plot tool now type-checks its React surface end to end with zero
`: any` destructures.

Counts retired per tool:

| Tool     | `: any` before | After |
| -------- | -------------: | ----: |
| Aequorin |            170 |     0 |
| Boxplot  |            160 |     0 |
| Heatmap  |            101 |     0 |
| Scatter  |             49 |     0 |
| Volcano  |             39 |     0 |
| Lineplot |             34 |     0 |
| UpSet    |              8 |     0 |
| Venn     |              6 |     0 |

The shape of the work was uniform across tools: every prop bag that the
chart, controls, steps, or plot-area files were destructuring as `: any` got
a typed interface declared alongside the pure helpers in
`<tool>/helpers.ts` (`BoxplotChartProps`, `VolcanoStyleTileProps`,
`HeatmapChartProps`, …). Discriminated unions replaced opaque-bag patterns
where they fit naturally — Heatmap's `ClusterResult` is now
`mode: "hierarchical" | "kmeans"`, Boxplot's `AnnotationSpec` includes a
`"both"` arm for facet+subgroup, Scatter's `computeLinearRegression` returns
`{ valid: false } | { valid: true; … }` so consumers narrow with
`if (rs.valid)` instead of optional-chaining at every field.

Latent bugs surfaced by typing:

- **Venn aria-label was reading `sets[name]` on a `Map`** (always
  undefined). Typing `sets: SetMap` made the bug visible; fix is one line.
- **Boxplot stats passed `null` through `if (!g.stats)` checks that
  TypeScript couldn't narrow** because the prop was typed `any`. Tightened
  to `BoxplotGroupStats | null`; chart consumers' guards now narrow.
- **Volcano `eligibleColumns` accepted an opaque bag**; now
  `ParseDataResult | null | undefined`, surfacing two callers that were
  passing the wrong shape.

CLAUDE.md's "Code style & conventions" section was also updated mid-cycle to
retire the prior `: any` blessing for sprawling step-component prop bags
and document the typed-interface pattern as the canonical approach.

## ➗ `tools/stats.js` carved into five focused files

The single 2566-line `tools/stats.js` was the largest file in the repo and
had grown into five distinct concerns: distributions and special functions;
sample tests; post-hoc procedures; clustering; multiset intersection. The
file was carved along those lines:

- **`stats-dist.js`** — distributions (norm / t / chi² / F / qtukey /
  noncentral cdfs), special functions (erf / gammaln / lbeta), `bisect`,
  power calculations.
- **`stats-tests.js`** — sample helpers (`sampleMean`, `sampleSD`),
  Shapiro-Wilk, Levene, Welch + Student t-test, Mann-Whitney U, one-way ANOVA
  (Welch + Fisher), Kruskal-Wallis, effect sizes.
- **`stats-posthoc.js`** — studentized range cdf, Tukey HSD, Games-Howell,
  Dunn, Benjamini-Hochberg, compact-letter-display, `selectTest`.
- **`stats-cluster.js`** — pairwise distance, hierarchical clustering, k-means.
- **`stats-msi.js`** — multiset intersection p-values for UpSet
  (`SuperExactTest` adaptation).

Crucially: every global name and every public function signature is
unchanged. The five files are concatenated into `shared.bundle.js` in the
right load order (`stats-dist` → `stats-tests` → `stats-posthoc` →
`stats-cluster` → `stats-msi`) so consumer code that names them as globals
keeps working with no edits. Test loaders now route through
`tests/helpers/stats-source.js`'s `STATS_FILES` list — one source of truth
instead of eight scattered `readFileSync` calls.

Each new file lands in the 200–800 LOC range that matches the rest of the
`shared-*.js` family and individual `_shell/` modules. The carve gives
mutation-testing and property-testing scopes a much sharper target than the
monolith allowed.

## 📊 Volcano label placement — smarter, multi-restart, sub-degree

The greedy-first-fit label layout that shipped in v1.2.0 had three known
shortcomings that surfaced regularly in dense gene-cluster screenshots:

1. **Forced labels defaulted to 12 o'clock** when no candidate slot was
   clean. In tight clusters this produced a visible vertical pile of
   labels right above the densest region.
2. **Greedy was order-sensitive.** The same input set in a different order
   would lock different high-priority labels into the easy slots, and the
   layout had no way to discover that.
3. **Coarse 24-angle × 4-distance grid.** No sub-degree precision; labels
   that needed a few degrees' lateral adjustment to escape an overlap
   couldn't find it.

Four targeted improvements:

1. **Forced fallback now scores every in-bounds candidate.** Weighted
   overlap penalty (text-on-dot > leader-vs-point ≈ leader-vs-leader >
   bbox overlap area) picks the least-bad. Replaces the 12-o'clock pile.
2. **K=4 input-order restarts.** The wrapper runs the greedy pass with
   four orderings (input order, reverse, most-isolated-first,
   most-clustered-first) and returns the result with lowest total
   penalty. Strictly non-regressing — the worst case is the original input
   order's output.
3. **Two-level fine sweep around forced winners.** Level 1 sweeps ±7.5°
   around the forced-fallback's winning angle; level 2 sweeps ±3.75°
   around level 1's winner. Labels that need a few degrees' adjustment
   can now find it.
4. **Density-aware cap warning in the Labels tile.** Chart now surfaces
   `(forcedCount, attemptedCount)` to the controls. When `forcedCount > 0`
   the tile shows _"N of M labels couldn't place cleanly"_ and a
   _"Use suggested"_ button that drops top-N to the actual clean budget.
   The cap is calibrated to the layout's real outcome, not a heuristic.

The fixed forced-12-o'clock from before is the obvious regression target
in screenshots; multi-restart + sub-degree close most of the rest.

## 🔬 Stats test/post-hoc dispatch consolidated behind one registry

Pre-registry, the same identifier → (function, label, post-hoc, k=2
membership) mapping was duplicated across:

- `_shell/stats-dispatch.ts` (the runtime dispatcher)
- `shared-stats-tile.js` (the StatsTile UI)
- `shared-r-export.js` (the R-script export)
- `aequorin/reports.ts` (Aequorin's per-condition stats reporter)

Eight string-matching sites. No compile-time guarantee they stayed in sync.
A new test added to `selectTest` would compile and run, but the StatsTile
might not know how to render its label, the R-script export might emit the
wrong call, and the Aequorin report might silently skip it.

The new `tools/_shell/stats-registry.ts` (originally
`tools/shared-stats-registry.js`, then migrated as part of cluster C) is the
single source of truth. Every consumer reads from `STATS_TEST_REGISTRY` /
`STATS_POSTHOC_REGISTRY` / `STATS_TESTS_FOR_K2` / `STATS_TESTS_FOR_K`. Adding
a new test now requires editing exactly one row instead of eight sites.
`tests/stats-registry.test.js` (22 assertions) pins routing parity with
direct `stats.js` calls and locks down the post-hoc / arity tables.

Pure refactor — no runtime behaviour change. The benchmark suite confirmed
output parity vs the pre-registry runtime.

## 🎨 Theme-var audit — second pass

The 2026-05-04 first-pass audit replaced bare-value chrome literals
(`background: "#fff"`, `boxShadow: "rgba(...)"`) with theme variables and
widened the `plottr/no-chrome-hex-literal` lint rule to catch named CSS
colours and `rgba`/`hsl` substrings. Two gaps remained:

1. **Multi-token hex shorthand** (`border: "1px solid #0072B2"`). The
   rule's HEX regex required a full-string match; widened to substring
   with hex-boundary guards.
2. **Ternary / `&&` / `||` / template-literal values inside `style={{...}}`**.
   The rule only inspected `Literal` property values; it now recurses
   through `ConditionalExpression`, `LogicalExpression`, and
   template-literal quasis.

With the widened rule, 17 fresh violations surfaced and were fixed —
aequorin parse-banner bg/border ternaries, boxplot facet-legend swatches,
molarity success-borders + table hairline, power calculator's accent-blue
chrome, scatter regression-info / shape-warning / "all"-button borders,
scatter palette-preview + venn region-table header borders. CSS-file gaps
the JS-only rule can't see were also closed by hand: four `box-shadow` rgb
literals in `tools/components.css` (`.dv-btn-plot:hover` /
`.dv-btn-secondary:hover` / `.dv-btn-dl:hover` / `.dv-btn-danger:hover`)
and the prefs-popover shadow in `tools/shared-prefs.js` now reference new
`--btn-*-hover-shadow` tokens (light: accent-tinted; dark: black-tinted
with deeper alpha, mirroring the established `--cta-primary-shadow` flip
pattern). Two remaining literals in `React.createElement`-style shared
`.js` files (which the JSX rule never sees) — `tools/shared-file-drop.js`'s
over-size error tint and `tools/shared-long-format.js`'s on-chip text
colour — also fixed manually.

## 🧪 Property-test expansion + Stryker mutation testing

Test infrastructure work that paired with the React-tier typing and stats.js
carve-up:

### Per-tool fuzz harnesses replaced by fast-check property suites

Each tool's `tests/fuzz/<tool>.fuzz.js` driver (and the weekly
`fuzz-release.yml` sweep) is gone; coverage now runs inside `npm test` via
`tests/<tool>.property.test.js`. fast-check replaces the bespoke fuzz
harnesses for three reasons:

- **Automatic shrinking** on failure — the seeded fuzz harnesses reported
  only the seed; reproducing required manually bisecting.
- **Routine cadence** — properties run on every `npm test` instead of being
  gated behind a separate command, so coverage can't drift between release
  sweeps.
- **Stronger invariants** — properties like _"summarize counts equal the
  result of classifying each point individually"_ or _"circleOverlapArea
  is symmetric in r1/r2"_ are clumsy in imperative fuzz-loop style.

### Property-test coverage expansion

117 → 247 tool-level properties (heatmap 35, boxplot 34, upset 27, venn 27,
aequorin 27, scatter 22, lineplot 18, volcano unchanged at 57). Plus new
distribution-level coverage:

- **`tests/stats.property.test.js`** — 48 properties (cdf monotonicity,
  cdf↔inv round-trips, gammaln recursion, swap-symmetry of two-sample
  tests, BH order preservation).
- **`tests/power.property.test.js`** — 45 properties (power ∈ [0, 1],
  null-hypothesis floor, monotonicity in n / es / α, 1-tail ≥ 2-tail,
  effect-size converter scaling laws).

Suite total: **1471 tests** (was 1218 at start of cycle).

### Stryker mutation testing wired up

`npm run mutation` runs Stryker against the files listed in
`stryker.conf.mjs`. Mutation testing meta-tests the test suite itself by
mutating source on disk and re-running `npm test` against each variant — a
_killed_ mutant means at least one test failed on the broken code; a
_surviving_ mutant points at an invariant the suite doesn't actually
constrain.

Files measured this cycle:

| File                        | Mutants | Score                                                                             |
| --------------------------- | ------: | --------------------------------------------------------------------------------- |
| `tools/volcano/helpers.ts`  |     996 | **100 %** (initial scope, all mutants killed)                                     |
| `tools/scatter/helpers.ts`  |      88 | **93.18 % raw / 100 % non-equivalent** (6 equivalent)                             |
| `tools/lineplot/helpers.ts` |     125 | **79.20 % raw / ~95 % non-equivalent** (~16 equivalent + ~7 contrived-input-only) |
| `tools/stats.js`            |    3504 | **14.87 % raw / 55.37 % covered** (the largest scope)                             |

The scatter and lineplot results required loader-pattern changes — Stryker's
per-test coverage instrumentation can't see source loaded via
`vm.runInContext`, so each tool's loader had to compile helpers via
`require()` from a temp `.cjs` file (scatter directly; lineplot as a hybrid
because its helpers reference shared globals as free variables). Five new
sharp boundary properties closed real gaps surfaced by each run.

The stats.js result is dominated by the no-coverage cohort — `tests/stats.test.js`'s
R cross-validation suite is skipped under Stryker because its inner
numerical loops slow ~3000× under perTest-coverage instrumentation. The
covered survivors cluster on Abramowitz & Stegun polynomial coefficients
where the property tests' tolerances are too loose to catch precision-shifting
mutations on the constants. Not a CI gate; mutation testing is a
quarterly exercise.

## 🧰 UX, layout, and chart fixes

A handful of user-visible fixes shipped alongside the refactor work:

- **Boxplot rotated x-tick labels no longer clip into the legend.** The
  bottom-margin reservation factored in only the rotation angle, not the
  label length. The reservation now takes the larger of the old
  angle-only heuristic and a sharper
  `maxLabelLen × charWidth × sin(angle)` estimate.
- **Lineplot / scatter / aequorin: long category names truncate in the
  legend.** The shared legend's fixed-width column had a ~13-char text
  budget; longer labels overflowed into the next column's icon. Boxplot
  already passed `truncateLabel = 14`; the other three now do the same.
- **Volcano: points at p=1 collided with the bottom x-axis line.** Auto
  y-range now opens a 3 % bottom pad below 0 for ~10–17 px of clearance.
  User-supplied `yMin` overrides remain unchanged.
- **Sidebar control-panel spacing was inconsistent across and within
  tools.** Two stacked bugs (boxplot's stray `marginBottom: 6` and
  `.dv-panel`'s baked-in `margin-bottom: 16px`) made the gap pulse
  between 10 px and 26 px depending on tile type. Fix at the shell
  level: `PlotSidebar` carries a `dv-sidebar` class, and `components.css`
  zeros `margin-bottom` on every `.dv-panel` direct child.
- **Power calculator was left-aligned instead of centered.** Missing
  `margin: "0 auto"`; now matches the molarity calculator.
- **Power calculator collapses to a single column with no plot card
  below 900 px viewport.** Mirrors molarity's `useIsMobile(900)` hook.
  The PowerCurve SVG is unreadable at that size.
- **Boxplot facets stretch to full content-column width.** The 720 px
  cap on non-subgrouped facets is gone; facets and the stats panel now
  follow viewport resizes together.
- **Boxplot stats panel gains 16 px breathing room above it**, matching
  the gap aequorin uses between its main plot card and the integral tile.

### Heatmap cell grid rasterizes via canvas

Cells used to render as N individual SVG `<rect>` elements (≈700 ms React
mount + 11 MB SVG for a 100 k-cell heatmap, with browsers paying
significantly more on top). They now paint to an off-screen canvas and ship
as a single PNG-encoded `<image>` inside `<g id="cells">`, dropping the
cell-grid cost to ~10 ms paint + ~10–30 ms encode and shrinking exported
SVGs by **~50×**.

Per-cell SVG `id="cell-<row>-<col>"` traceability is gone; every other named
group (`#col-dendrogram`, `#row-dendrogram`, `#cells`, `#colorbar`,
`#col-labels`, `#row-labels`, `#selection-mask`) is unchanged. Hit-testing,
hover tooltip, brush selection, and dendrograms are all unaffected — they
were already coordinate-based, not per-rect events.

### Tukey HSD pair-coverage property tightened

The `tukeyHSD` pair-coverage fast-check property occasionally hit 30 s
timeouts at `k = 6` groups with `n = 2` per group — qtukey is a 200-step
bisection over a 48-node Gauss-Legendre quadrature of ptukey at df = 1
(~10s of ms per pair × 15 pairs × 80 runs blew past the per-test budget).
Capping at `k ≤ 4` and requiring `n ≥ 3` keeps the workload well below the
budget without weakening the structural invariant — pair coverage doesn't
depend on group count beyond k ≥ 2.

## 🧹 Removed

- **Heatmap detail-view dendrogram thin/medium/bold selector.** The
  toggle had no real workflow value — the detail plot's row pitch is
  already enlarged for readability, so the stroke-width choice didn't
  change what users could see. Stroke now hard-codes to the prior
  "medium" default (1.5 px). The detail-view header simplifies to just
  the SVG / PNG download buttons.
- **Landing-page footer ("N internal tests / N R cross-checks /
  view benchmark →").** The numbers were repository trivia rather than
  user-facing information; the benchmark count is on `benchmark.html`,
  and the "view benchmark →" link duplicated the trust-badge pill at
  the top. The "N internal tests" badge is gone from the trust-badge
  `title=` attribute too. `scripts/bump-test-count.js` and
  `scripts/run-vitest.js` retired (the wrapper only existed to feed
  the bumper); `npm test` is now just `vitest run`. CI's
  _Verify landing-page test-count badge_ step deleted along with the
  artefact it gated.
