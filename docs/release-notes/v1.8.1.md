# Plöttr v1.8.1 "Zip it" — release notes

A focused download-UX release. Saving several files at once now behaves
predictably across browsers, and a large export comes down as a single tidy
`.zip` instead of a flurry of individual downloads.

The name is the obvious one: past three files, Plöttr zips it.

## 🐛 Fixed

### Multi-file downloads no longer scatter

Several tools can emit more than one file from a single "download" click — a
faceted Group Plot or RLU time-course exports one SVG/PNG per facet, Venn and
UpSet can dump one CSV per region/intersection, and the box/line/scatter stats
panels write one report (and one R script) per group. Previously each of those
files was saved by its own `showSaveFilePicker` call fired in the same tick.

Only the **first** call holds the browser's user-gesture activation, so on
recent Chromium (notably Windows) the first file opened a Save dialog and every
later call threw `NotAllowedError` and fell through to the classic
`<a download>` anchor — silently routing the rest into the default Downloads
folder. The result: the first file landed where you chose, the others landed
somewhere else. The older `setTimeout`-staggered loops in Venn/UpSet didn't
help, because a deferred callback has no user activation at all.

Saving is now tiered by file count, in `_core/download.ts`'s `saveBlobs`:

- **1 file** → the familiar Save As dialog (unchanged).
- **2–3 files** → one `showDirectoryPicker` prompt; every file is written into
  the folder you choose.
- **More than 3 files** → bundled into a single `.zip` and saved as one file.

Single-file flows are untouched. The directory-picker tier falls back to
staggered anchor downloads on Firefox / Safari (which have no directory
picker); the zip tier needs no File System Access API at all — a zip is just
one download — so a large export comes down cleanly on every browser.

### In-browser zip, no new dependency

The archive is built by a new, self-contained `_core/zip.ts` (`buildZip`) rather
than a vendored library — matching the project's no-runtime-dependency stance.
It writes a classic ZIP: method 0 / STORE (no compression), a CRC-32 per entry,
UTF-8 filenames, and no ZIP64 records (every Plöttr export sits far below the
4 GB / 65 535-entry limits). STORE keeps the format trivially correct on every
unzip tool and avoids depending on `CompressionStream`; the exports it bundles
(small CSV / SVG / short text reports) wouldn't compress to much anyway. The
archive is named after the common prefix of the files it carries (e.g. a faceted
export → `…_groupplot_Day.zip`, Venn regions → `…_venn.zip`), with
collision-safe entry names.

`tests/zip.test.js` round-trips the output through a minimal STORE reader,
asserting entry order, names, byte-exact contents, sizes, CRC-32, the UTF-8
filename flag, and a full 0–255 byte-range payload.
