# Plöttr v1.0.4 — release notes

_Released 2026-05-02._

A licensing / paperwork release: a real `LICENSE` file lands at the repo
root, every algorithmic / vendor attribution is consolidated into
`THIRD_PARTY.md`, and the three continued-fraction primitives that had
quietly transliterated _Numerical Recipes in C_ are swapped for
public-domain Cephes ports. The benchmark page's brand drift and the
RLU-timecourse axis-input unit bug are also fixed.

## Added

**License paperwork in place for Zenodo deposit.** A `LICENSE` file at the repo root (MIT, "Copyright (c) 2026 Plöttr contributors"), a `vendor/LICENSE-react.txt` carrying the full upstream MIT text for the vendored React + ReactDOM (Meta) — alongside the inline `@license React` headers already in the bundles — and a top-level `THIRD_PARTY.md` consolidating every algorithmic reference and third-party attribution that lives inline in source comments today (Acklam for `norminv`, Royston 1995 / AS R94 for Shapiro-Wilk, A&S 26.2.17 for `normcdf`, Cephes 2.8 / Moshier for the continued-fraction primitives, Lanczos g = 7 / Godfrey 2001 for `gammaln`, Park-Miller 1988 for the LCG, Wong 2011 for the Okabe-Ito palette, Heckbert 1990 for the nice-numbers tick rounding, Silverman 1986 for KDE bandwidth, Day & Quinn 1989 for Games-Howell, Lehmann 1975 / Siegel-Castellan / Tomczak 2014 / Kerby 2014 for Mann-Whitney / Kruskal-Wallis tie corrections and effect sizes, Piepho 2004 for compact letter display, Wang/Zhao/Bhattacharya 2015 for multi-set intersection enrichment, Lance-Williams 1967 for hierarchical clustering, Lloyd 1982 / Arthur-Vassilvitskii 2007 for k-means / k-means++, viridis & cividis colour-map provenance). README's `Citing` section gained a `License` sibling so the legal posture is one click from the front page. `package.json` now declares `"license": "MIT"` for tooling that reads it.

## Changed

**`betacf`, `gammainc`, `gammainc_upper` swapped to Cephes-derived ports (public domain).** A licensed-source audit flagged the prior implementations of these three continued-fraction / series-expansion primitives as near-verbatim transliterations of _Numerical Recipes in C_ (Press et al., §§6.2 / 6.4) — same `qab`/`qap`/`qam` variable names, same `1e-30` Lentz floor clamp, same `eps = 3e-14` convergence test, same per-iteration alternation structure. NR has a restrictive license; an unattributed port of its _implementation choices_ (not the algorithm — Lentz's method itself is in the open literature) is a real legal risk for a Zenodo deposit. Replaced with ports of `incbcf` and `igam` / `igamc` from the [Cephes Mathematical Library](https://www.netlib.org/cephes/) (Stephen L. Moshier, "Cephes Math Library Release 2.8", 2000), which the author dedicated to the public domain. Cephes uses a structurally distinct three-term recurrence (`pkm1`/`pkm2`/`qkm1`/`qkm2`) with periodic `big`/`biginv` rescaling instead of Lentz's per-step floor clamp; the two converge to the same value within machine precision on the inputs we use. The Plöttr-specific polish (log-space final exponentiation, √a-scaled iteration cap so `chi2cdf` / `ptukey` stay accurate at huge df) is layered on top of the Cephes recurrence. Cross-validated against R via `npm run benchmark`: **299/303 comparisons pass, 0 fail, 4 past R's floor** (cases where JS continues the true tail beyond R's `1 − ptukey(q)` cancellation point — same behaviour as before, this is the audit-22 #13 deep-tail accuracy work). Top deep-tail drifts: 3.16e-2 log-space (≈ 3% relative) on a Games-Howell pAdj at p ≈ 1.4e-8; absolute deltas are 1e-10 to 1e-12 — well inside the existing `P_LOG_TOL = log(1.1) ≈ 9.5%` benchmark tolerance. All 891 deterministic tests still pass. The drift is the natural numerical difference between the Cephes recurrence and modified Lentz on deep-tail p-values where small absolute differences amplify in log-space; well within the in-repo statistical tolerance bar.

## Fixed

**Benchmark page: collapsible categories restored, "Toolbox" / "dataviz" labels renamed to "Plöttr".** Two related drifts on `benchmark.html`, both rooted in the same anti-pattern — fixes landed on the _generated_ HTML instead of the generator (`benchmark/run.js`), so every `npm run benchmark` run reverted them. (1) Per-category sections are now `<details class="category" open>` wrappers with the heading + pass/fail badge inside `<summary>`; CSS-only chevron rotates 90° when open, default `<details>` marker hidden via `list-style: none`. Default state open, so the page reads identically to before until the user clicks to collapse. No JavaScript needed. (2) The Apr-22 project rename (`5528471 rename: dataviz → Plöttr`) edited the rendered `benchmark.html` `<title>` only and never touched the generator; the per-category table column header `Toolbox` was never renamed at all (still original from the first benchmark commit). Now in the generator: `<title>` is "Plöttr · statistical benchmark vs R", every per-category table column header reads "Plöttr", and the prose "Inputs are bit-identical between R and the toolbox" reads "...between R and Plöttr". Internal `dataviz-theme` localStorage key + `dataviz-theme-change` DOM event + `dataviz-theme-set` postMessage type are deliberately untouched (renaming would invalidate every existing user's saved theme + per-tool auto-prefs — same rationale as the original rename commit).

**README — benchmark count corrected and `max |Δ|` clarified as absolute.** `293 comparisons` → `303` (the prior count was stale; cached `results-r.json` grew). The `8 × 10⁻⁷` figure is unchanged — verified by recomputing `max |R − JS|` across all 303 rows after the Cephes port. README now explicitly frames it as the **absolute** max across all comparisons (the worst case lives on a Dunn (BH) pAdj at p ≈ 0.0125 with relative drift 0.006%), with the test-statistics-only max (4 × 10⁻⁹ on Shapiro-Wilk W) added for reviewers who care about W/t/F/H accuracy independent of p-value underflow regime. The `npm run benchmark` summary's `max |Δ| = 3.16e-2` is log-space — easy to misread as a hard accuracy claim.

**RLU timecourse — X start / X end inputs now follow the Display unit.** `vis.xStart` and `vis.xEnd` are stored as row indices (integers used to slice calData), and the chart correctly multiplies them by `timeStep * convFactor` to render the X axis in display units. The input fields in the Axes control panel skipped the conversion: switching Display unit from e.g. `s` to `min` re-rendered the chart correctly but left the inputs showing the raw row-index number, so the user couldn't type a coherent range in the new unit. The inputs now apply the same `* timeStep * convertTime(base→display)` round-trip the chart uses, and the labels carry the active unit (`X start (min)` / `X end (s)` / etc.) so it's obvious which scale to type in. Storage stays as row indices — no migration needed for saved auto-prefs, no change to the row-slicing logic at `tools/aequorin/index.tsx:179`.
