# Plöttr v1.0.5 — release notes

_Released 2026-05-02._

A small UX-and-polish release: the in-app sidebar tile titles finally read
the same across every plot tool, the suggested citation drops the unjustified
"plant scientists" framing, and a handful of accessibility / visual fixes
land alongside (RLU timecourse layout collisions on narrow columns, the
auto-Y-range first-render glitch, dark-mode favicon mismatch, link-contrast
holes on the privacy page and landing footer).

## Changed

**Control-tile titles unified to the heatmap convention across every plot tool.** Heatmap's sidebar tile titles ("Display", "Style", "Cluster", …) rendered in 12 px uppercase semibold with 0.5 px letter-spacing — visually distinct, easy to scan, the canonical look. Every other tool drifted to plain mixed-case 12 px or 13 px regular — same control, less prominent, inconsistent across pages. The shared `ActionsPanel` was the most obvious offender: every plot tool's sidebar showed a soft-spoken "Actions" tile while everything around it used different conventions. Introduced a `.dv-tile-title` class in `tools/components.css` (`font-size: 12px; font-weight: 600; color: var(--text-muted); text-transform: uppercase; letter-spacing: 0.5px; font-family: inherit`) as the single source of truth, then swept every site so they all read the same: **(a)** the six per-tool `ControlSection` collapsible-header definitions (heatmap, aequorin, boxplot, lineplot, scatter, upset); **(b)** the shared `ActionsPanel` "Actions" title in `tools/shared-ui.js`; **(c)** every static plot-step tile title — venn ("Sets", "Display"), scatter ("Variables", "Regression line", "Filters"), upset ("Data format", "Sets to include", "Intersection cutoff", "Intersection significance"), lineplot ("Variables"). At each call site the duplicate `fontSize` / `fontWeight` / `color` / `textTransform` / `letterSpacing` were dropped from the inline `style={…}` (inline styles win against external rules — leaving them in would have kept the old look), so `.dv-tile-title` is now the only definition of tile-title typography. Two upset tiles ("Sets to include", "Intersection cutoff") had drifted to `var(--text)` (full text colour) instead of the canonical `var(--text-muted)`; both now match. Step-navigation chrome, configure-step subheads (Groups / Assumptions / Test in stats panels, ColumnRoleEditor headings, etc.), and the molarity / power calculator panels are intentionally **out of scope** — they're a different surface with their own tier in the typographic hierarchy. No JS change, no `vis` migration; pure CSS-and-className refactor.

**Suggested citation now describes Plöttr as a toolbox "for wet-lab scientists" rather than "for plant scientists".** The product never had any plant-specific code paths — `tools/stats.js` and the seven plot tools are domain-neutral (the `tools/aequorin/helpers.ts` calibration formula is the only plant-aequorin-specific piece, and even that is a single optional pipeline inside the RLU timecourse tool). The narrower phrasing in `README.md`'s suggested-citation block undersold the audience and hid the tool from the broader wet-lab community it was always intended to serve. Single-line edit; the only occurrence of the phrase in the repo. Provenance comments inside `tools/aequorin/helpers.ts` (e.g. "treatment adopted for plant aequorin") are scientific scope notes about the calibration formula and stay unchanged.

**Benchmark page — third-person voice, tighter lede, consistent disclosure chrome, expand/collapse-all toggle.** `benchmark.html` previously narrated in first-person plural ("we compare", "our `ptukey_upper`", "we have work to do"), inconsistent with the rest of the product, which never says "we"; rewrote the lede in third-person ("Plöttr reruns…", "Plöttr's `ptukey_upper`…") and trimmed the long inline parentheticals about scipy / Monte-Carlo cross-checks and the prior self-referential post-hoc check, leaving one short claim per bullet — full provenance still lives in source comments for anyone digging into `tools/stats.js`. The per-category `<details>` summary used a bespoke CSS triangle (`border-left: 6px solid var(--text-muted)` + 4px transparent caps) that didn't match the canonical `.dv-disclosure` chevron used by StatsTile, Aequorin, and Scatter; `benchmark.html` now links `tools/components.css` and uses the standard `<span class="dv-disclosure">` glyph (soft neutral disc + masked SVG chevron, rotates 90° via `.category[open] > summary > .dv-disclosure`) so collapsibles look identical product-wide. Added an Expand all / Collapse all button below the summary card, styled to match the page chrome (border, hover-accent border + text); a single click flips every category, and the label re-syncs whenever an individual `<details>` is toggled (any closed → "Expand all", all open → "Collapse all"). All edits live in the generator (`benchmark/run.js`) so subsequent `npm run benchmark` runs reproduce the new copy and chrome verbatim — same anti-pattern guarded against in the 1.0.4 fix.

## Fixed

**RLU timecourse — Series / Layout toggles no longer collide with the Sample-selection pill or the chart on narrow columns.** The plot-step right column anchors two top-row widgets: the sticky "🔬 Sample selection" pill at top-left (kept visible while scrolling) and the absolute-positioned **Series** (Pool by name / Individual) + **Layout** (Combined / Faceted) toggle group at top-right (intentionally non-sticky so it scrolls away with the page — the differential stickiness is part of the design). At the desktop floor (~1100 px viewport) and slightly above, the chart column ends up around ~620 px wide; the sample pill (~175 px) plus the two toggle groups (~175 + 18 + 150 ≈ 343 px) plus inter-group breathing room exceeds that, and the absolute right group visually crashed into the sticky left pill. Fixed via a CSS container query on a new `.dv-aeq-chart-area` wrapper + `.dv-aeq-toggles-group` class in `tools/components.css`: when the chart column's inline-size drops below 620 px, the toggles **(a)** flip from a horizontal row (`gap: 18px`, `align-items: flex-start`) to a right-aligned vertical stack (`flex-direction: column`, `align-items: flex-end`, `gap: 8px`), **and (b)** drop their absolute positioning (`position: static; margin-bottom: 8px`) so they re-enter normal flow and push the chart below them downward — otherwise the now-taller vertical stack would overlap the top-right of the plot tile. Wide-screen layout is byte-identical to before; the differential stickiness contract is preserved (Sample-selection still sticky, Series + Layout still scroll away normally — only their internal axis flips and they take in-flow space). No JS change, no `vis` migration; a pure responsive-layout fix scoped to the aequorin/RLU-timecourse tool, with all positioning consolidated in `components.css` so the inline `style={…}` no longer fights the container query.

**RLU timecourse — first-render auto-Y glitch eliminated; range computation extracted as a pure helper.** When `vis.autoYRange` was on, the chart used to flicker once on first render — paint with the stale `vis.yMin` / `vis.yMax` (rehydrated from `dataviz-aequorin-prefs` of a previous session, or the `VIS_INIT_AEQUORIN` defaults `0.1` / `1.4`), then snap to the correct range one frame later. Root cause: the auto-rescale lived in a `React.useEffect` at `tools/aequorin/index.tsx:212` that fires _after_ the browser paints, so the wrong frame was visible to the user. If the saved range belonged to a wildly different dataset (different formula, different scale of data) the glitch was severe — chart axis ticks and data line were both visibly off until the effect committed. Fixed by switching to `React.useLayoutEffect`, which runs after DOM mutation but **before** paint; the resulting `updVis` re-render is flushed in the same paint cycle, so the user only ever sees the corrected range. No behaviour change when `autoYRange` is off (manual Y bounds still stick). The math itself was correct all along — extracted into a pure `computeAutoYRange(calData, xStart, xEnd)` helper in `tools/aequorin/helpers.ts` so it has a unit-test footprint independent of React lifecycle: 8 new tests in `tests/aequorin.test.js` pin the ±10 % padding contract, the `max(0, …)` clamp on `yMin`, the row-window honour, null/NaN/Infinity skipping, the empty-data short-circuit, the 2-decimal rounding, the xStart/xEnd clamping to `[0, calData.length-1]`, and the regression-specific guarantee that the helper depends only on the data — never on whatever `vis.yMin` / `vis.yMax` the React layer happens to be holding (the original bug surfaced precisely because that decoupling didn't exist). Total deterministic tests 891 → 899; landing-page badge bumped accordingly.

**Privacy page — "GitHub" link in the "Open to scrutiny" trust-card now follows site link conventions.** The third trust-card on `privacy.html` ("Plöttr is open source. The whole codebase — including this page — is on GitHub. Read it, fork it, or run a local copy.") had no `.trust-card a` rule, so the inline `<a href="https://github.com/evompmi/plottr">GitHub</a>` fell back to the UA-default `#0000EE` blue — same WCAG-AA contrast failure pattern fixed for the landing-page footer "view benchmark →" link in the previous accessibility pass. Added `.trust-card a` mirroring the convention used by `.inspect-block a` on the same page (`color: var(--accent-primary)`, no underline by default, underline on hover) so all three card-body links theme correctly and stay legible in both modes. No markup change.

**Landing page — accessibility, contrast and consistency pass.** A UI review surfaced three WCAG-AA contrast failures and several smaller inconsistencies on `index.html`. (1) Footer "view benchmark →" link inherited the UA-default `#0000EE` blue (no rule pointed at it), giving 1.97:1 against the dark `--page-bg` — now styled via `.landing-header p a` with `var(--text-muted)` + underline + `var(--accent-primary)` hover, mirroring `.trust-badge` chrome. (2) The UpSet tile icon hard-coded `#333` for the dot matrix, which is invisible (1.36:1) on dark `--surface`; switched the matrix dots and connector lines to `currentColor` (filled / `fill-opacity="0.25"` for "absent") so they pick up `--text` and theme cleanly — the topbar variant of the same icon already used this pattern. (3) `.tile-desc` used `var(--text-faint)` at 10 px (3.54:1 light / 3.89:1 dark, both fail AA for body text); promoted to `var(--text-muted)` (10.5:1 / 7.46:1, comfortably above AA). (4) Theme toggle was positioned with the magic-number `right: calc(50% - 157px)` to align with the 314 px tile grid — silently desyncs if tile widths change; replaced with `right: 16px` for a stable viewport-anchored position. (5) Casing normalised: `RLU timecourse` → `RLU Timecourse` and `UpSet plot` → `UpSet Plot`, both in the landing tile labels and in `TOOL_META`, so all nine tile labels are now Title Case. (6) Tagline ASCII bookends `- Paste, plot, publish -` swapped for proper em-dashes, matching the typographic care elsewhere on the page (`→` arrows, `±`, etc.). (7) Footer stats run-on (one `<p>` with `<br>`s) split into three discrete `<p>`s so screen readers don't read "891 internal tests 293 cross-checks vs R 4.5 view benchmark" as a single sentence. (8) `.tile` gained a `:focus-visible` outline (2 px solid `--accent-primary`, 2 px offset) matching `.tb-icon-btn` and `.theme-toggle-btn` — the tiles previously fell back to the UA default outline. (9) Dropped the `letter-spacing: 0.5px` on `.landing-header h1` — letter-spacing on a monospace font is a no-op since each glyph already has fixed advance.

**Browser favicon now matches the brand mark.** The favicon was a four-blue-dot abstract scatter that didn't match anything else in the product; replaced with the inline Ansuz rune SVG used in the `landing-header h1` `.brand-rune` (circle outline + three diagonal strokes), encoded as a `data:image/svg+xml,…` URL with `%23648FFF` for the brand blue. Same artwork in the page header and the bookmark/tab icon — one consistent identity.
