# Plöttr v1.9.1 "True Colours" — release notes

A focused correctness release. A custom regression-line colour you pick in the
Scatter tool now stays put when you change the line width, instead of snapping
back to the default red — and the underlying class of bug is hardened across
the per-series colour and reference-line controls in four tools.

The name is the obvious one: your colours now hold true.

## 🐛 Fixed

### Scatter regression-line colour no longer reverts on width change

After picking a custom trend-line colour and then dragging the width slider, the
colour jumped back to the default red (`#dc2626`). The cause was a stale-closure
interaction:

- `SliderControl` (`tools/_shell/SliderControl.tsx`) is wrapped in `React.memo`
  with a comparator that **deliberately ignores `onChange`**, so dragging one
  slider doesn't re-render every other slider in the same sidebar. The
  comparator's safety contract — spelled out in its header comment — is that
  every call site must close its inline `onChange` over a _stable_ setter plus a
  literal patch, with no captured state that can go stale.
- Scatter's regression width slider broke that contract. Its setter,
  `updRegression`, was a plain (non-memoised) arrow that closed over the
  `regression` snapshot: `updVis({ regression: { ...regression, ...patch } })`.

So the sequence was: you change the colour → the reducer updates → `App`
re-renders with a fresh `updRegression` bound to the new colour → but the
memoised slider sees `value`/`min`/`max`/`label` unchanged (and ignores
`onChange`), so it **skips re-rendering** and keeps its original `onChange`,
which is still bound to the _original_ `updRegression` over the _original_
`regression` (colour = red). Dragging width then replayed
`{ ...{ color: red }, strokeWidth }` and clobbered the colour.

The fix makes `updRegression` a stable `useCallback` that merges into the
_current_ reducer state via a functional updater, so the slider's retained
closure can never replay a stale snapshot:

```js
const updRegression = useCallback(
  (patch) => updVis((s) => ({ regression: { ...s.regression, ...patch } })),
  [updVis]
);
```

### The same trap, hardened everywhere it could recur

The shared `vis` reducer in `tools/_shell/usePlotToolState.ts` now accepts a
functional patch `(prev) => Partial<TVis>` in addition to a plain object and the
`_reset` sentinel — an additive, backward-compatible change. With that in place,
every per-series / per-category colour setter and reference-line setter that
previously closed over a `vis.*` snapshot was converted to read fresh state and
dropped its stale dependency:

- **Scatter** — `updRegression`, `setColorMapDiscrete`, `setSizeMapDiscrete`,
  `setShapeMapDiscrete`, `setRefLines`
- **Group Plot** — `setBoxplotColors`, `setCategoryColors`
- **Line Plot** — `setGroupColor`
- **Venn** — `setSetColors`

Only the regression-width path was user-visibly broken today (it was the one
setter handed to a memoised control that ignores `onChange`). The others were
latent: they happened to escape the bug only because their consuming components
re-render on each edit and hand back a fresh closure. Converting them removes the
footgun before a future memoised control reintroduces it, and as a side benefit
the setters are now genuinely stable references rather than churning on every
state change.

No behavioural change beyond the bug fix; the full deterministic test suite
(1834 tests) passes unchanged.
